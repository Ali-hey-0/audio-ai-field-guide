## 🧠 مدل ذهنی اصلی

پیش از ورود به جزئیات هر واحد، این پایپلاین شاخه‌ای را در ذهن حک کنید. **تقریباً هر سیستم یادگیری ماشین صوتی روی کره‌ی زمین، یکی از این دو مسیر است:**

```
                              ┌───────────────────────────┐
                              │      هوش مصنوعی صوت        │
                              └─────────────┬───────────────┘
                                            │
                  ┌─────────────────────────┴─────────────────────────┐
                  ▼                                                    ▼
       صوت ← مدل ← متن / برچسب                              متن ← مدل ← صوت
      ────────────────────────────────                     ────────────────────────
       تحلیلی / درک‌محور                                      تولیدی / سنتزمحور
       • ASR (گفتار به متن)                                  • TTS (متن به گفتار)
       • دسته‌بندی صوت                                        • تولید صوت
       • تشخیص گوینده                                        • شبیه‌سازی صدا (Voice Cloning)
       • تشخیص احساس
```

و زیر *هر دو* شاخه، یک اسکلت پنج‌مرحله‌ای یکسان تکرار می‌شود که فقط لبه‌های آن تغییر می‌کند:

```
سیگنال خام ← پیش‌پردازش ← استخراج ویژگی ← بدنه‌ی ترنسفورمر ← سر وظیفه ← پیش‌بینی
```

اگر از قبل با ترنسفورمرهای NLP (مثل BERT یا GPT) آشنا هستید، این نگاشت مستقیم را ببینید — این شاید مفیدترین قیاس در کل این حوزه باشد:

| حوزه | ورودی خام | تبدیل به | سپس تغذیه به | سر خروجی |
|---|---|---|---|---|
| **NLP** | متن | توکن‌ها ← Embedding | Encoder/Decoder ترنسفورمر | سر پیش‌بینی توکن بعدی / دسته‌بندی |
| **صوت** | موج صدا (اعداد اعشاری) | ویژگی‌های یادگرفته‌شده‌ی CNN یا Mel Spectrogram ← Embedding | Encoder/Decoder ترنسفورمر | سر کاراکتر/برچسب/Mel-spectrogram |

**حدود ۸۰٪ از چیزی که از قبل می‌دانید، مستقیماً قابل انتقال است.** تنها بخش واقعاً جدید این ماجرا این است که *سیگنال خام چگونه ابتدا به یک دنباله از بردارها تبدیل می‌شود* — هرچه بعد از آن می‌آید (self-attention، تفکیک encoder/decoder، سرهای وظیفه، استراتژی فاین‌تیون) همان داستان قبلی است، فقط با اسم‌های صوتی به‌جای اسم‌های متنی.

## واحد ۱ — کار با داده‌های صوتی

### انتزاع کتابخانه‌ی `datasets`

```bash
pip install datasets[audio]
```

```python
from datasets import load_dataset
dataset = load_dataset("PolyAI/minds14")   # قصدهای مکالمات مرکز تماس بانکی؛ مثال اصلی این دوره
example = dataset[0]
```

نکته‌ی کلیدی و به‌راحتی قابل‌اغماض این است: **ستون `audio` یک مسیر فایل نیست — یک نوع ویژگی هوشمند و تنبل (lazy) است.** هر نمونه به یک ساختار ثابت تجزیه می‌شود:

```
نمونه
  ├── audio
  │     ├── path            → مسیر فایل روی دیسک
  │     ├── array            → خود موج صدا: [0.02, -0.03, 0.11, ...]
  │     └── sampling_rate    → مثلاً ۱۶۰۰۰ (۱۶٬۰۰۰ نمونه در ثانیه)
  ├── transcription          → متن برچسب‌گذاری‌شده توسط انسان (حقیقت پایه‌ی ASR)
  └── intent / label / speaker / emotion   → هدف اختصاصی همان وظیفه
```

**نمونه‌ای از یک سطر در MINDS-14:**

| audio | transcription | intent |
|---|---|---|
| `audio1.wav` | "Pay my bill" | `PAY_BILL` |
| `audio2.wav` | "Check balance" | `BALANCE` |

خود فیلد `intent` به‌صورت یک شناسه‌ی عددی کلاس ذخیره می‌شود (مثلاً `13`)، نه یک رشته‌ی قابل‌خواندن — `dataset.features["intent_class"].int2str(13)` آن را به `"PAY_BILL"` برمی‌گرداند. این الگو (ذخیره به‌صورت عدد، رمزگشایی با یک تابع کمکی) در سراسر دیتاست‌های دسته‌بندی Hugging Face استاندارد است، نه چیزی مخصوص MINDS-14.

> **نکته‌ای که برای افراد آشنا به NLP/بینایی ماشین گیج‌کننده است:** اینجا معادلی برای «فقط عکس رو باز کن و نگاهش کن» وجود ندارد. مدل — و شما هنگام دیباگ کردن — فقط و فقط `audio["array"]` را می‌بینید، یعنی یک آرایه‌ی اعشاری تک‌بعدی ساده. خود فایل WAV بعد از دیکود شدن دیگر هرگز لمس نمی‌شود.

### ساختار جهانی دیتاست (چرا این ساختار در کل هوش مصنوعی صوت تعمیم پیدا می‌کند)

```
نمونه ← Audio ← موج صدا ← برچسب
```

فقط **برچسب** بین وظایف مختلف تغییر می‌کند:

| اگر برچسب... باشد | وظیفه... است |
|---|---|
| متن پیاده‌شده (Transcription) | ASR |
| نام قصد / کلاس | دسته‌بندی صوت |
| شناسه‌ی گوینده | تشخیص گوینده |
| برچسب احساس | تشخیص احساس گفتار |

این *دقیقاً* دلیل این است که نوع ویژگی `Audio` در Hugging Face مستقل از وظیفه است — همان کد بارگذاری، دیکود کردن و تغییر نرخ نمونه‌برداری، چه بخواهید Whisper آموزش دهید و چه یک مدل تشخیص گوینده، بدون تغییر کار می‌کند. لایه‌ی زیرساخت اهمیتی نمی‌دهد که دارید چه چیزی را پیش‌بینی می‌کنید.

### پیش‌پردازش: تغییر نرخ نمونه‌برداری، مرحله‌ای است که نمی‌توانید از آن بگذرید

دیتاست‌های واقعی با نرخ‌های نمونه‌برداری ناهماهنگ می‌رسند:

```
صوت A ← ۱۶ کیلوهرتز،  ۲ ثانیه
صوت B ← ۴۴٫۱ کیلوهرتز، ۸ ثانیه
صوت C ← ۴۸ کیلوهرتز،  ۱۵ ثانیه
```

اگر این‌ها را مستقیماً به مدل بدهید، رفتار نادرستی نشان می‌دهد — بیشتر ترنسفورمرهای گفتاری (Whisper، Wav2Vec2) روی **۱۶ کیلوهرتز** آموزش دیده‌اند، و ویژگی‌های فرکانسی در نرخ‌های دیگر با چیزی که مدل واقعاً یاد گرفته مطابقت ندارد. برخی مدل‌ها به‌سادگی ورودی را رد می‌کنند؛ برخی دیگر به‌طور نامحسوس افت کیفیت پیدا می‌کنند.

**راه‌حل — ستون را cast کنید، کتابخانه به‌صورت تنبل و در لحظه‌ی دسترسی تغییر نرخ می‌دهد:**

```python
from datasets import Audio
dataset = dataset.cast_column("audio", Audio(sampling_rate=16_000))
```

```
Audio قدیمی ← خواندن مجدد ← تغییر نرخ ← Audio جدید     (به‌طور شفاف در هر بار دسترسی رخ می‌دهد)
```

با بررسی `sampling_rate` قبل و بعد از عملیات، صحت آن را تأیید کنید — `44100 ← 16000` نشانه‌ی موفقیت است. **تعداد نمونه‌ها** نیز به همان نسبت تغییر می‌کند (۴۴٬۱۰۰ نمونه در ثانیه از صوت ← ۱۶٬۰۰۰ نمونه در ثانیه از *همان* صوت)، اما مدت زمان کلی پخش و پرده‌ی صوتی (pitch) بدون تغییر باقی می‌مانند.

> **نکته‌ی غیربدیهی که ارزش به‌خاطر سپردن دارد:** تغییر نرخ نمونه‌برداری، *چند عدد نماینده‌ی یک ثانیه از صدا هستند* را تغییر می‌دهد — سرعت یا پرده‌ی صدا را تغییر **نمی‌دهد**. این رایج‌ترین سوءتفاهم مفهومی برای تازه‌واردهای صوت دیجیتال است: تغییر نرخ نمونه‌برداری با کش/فشرده کردن زمانی (time-stretching) یکی نیست. یک کلیپ ۲ ثانیه‌ای، بعد از تغییر نرخ همچنان ۲ ثانیه است؛ فقط حالا با اعداد کمتر (یا بیشتری) توصیف می‌شود.

**چرا نرخ‌های ناهماهنگ واقعاً به مدل آسیب می‌زنند**، نه فقط «اشتباه به نظر می‌رسند»:
- ویژگی‌های فرکانسی که مدل یاد گرفته تشخیص دهد، در موقعیت‌های نمونه‌ی مطلق متفاوتی از انتظار رخ می‌دهند.
- عملکرد حتی وقتی محتوای صوتی یکسان است نیز افت می‌کند.
- برخی معماری‌ها به‌جای افت تدریجی، ورودی را کاملاً رد می‌کنند.

### چک‌لیست کامل پیش‌پردازش (فراتر از آنچه یک درس به‌تنهایی پوشش می‌دهد)

واحد درسی فقط روی تغییر نرخ نمونه‌برداری تمرکز می‌کند، اما پایپلاین‌های تولیدی معمولاً موارد زیر را هم اضافه می‌کنند:

- ✅ تغییر نرخ به نرخ مورد انتظار مدل
- ✅ نرمال‌سازی بلندی صدا (Loudness)
- ✅ حذف سکوت‌های اضافی
- ✅ کاهش نویز
- ✅ بالشتک‌گذاری (Padding) یا کوتاه‌سازی به طول ثابت (برای batch کردن)
- ✅ استخراج ویژگی — Mel Spectrogram یا MFCC، اگر معماری به ویژگی‌های دستی نیاز داشته باشد نه یادگیری ویژگی‌های خودش

```
فایل صوتی ← خواندن صوت ← بررسی نرخ نمونه‌برداری ← تغییر نرخ (مثلاً ← ۱۶ کیلوهرتز) ← آماده برای مدل
```

### استریمینگ: وقتی دیتاست روی دیسک جا نمی‌شود

```python
dataset = load_dataset("...", streaming=True)
```

**مشکلی که این روش حل می‌کند:** یک دیتاست ۲ ترابایتی را تصور کنید. جریان استاندارد —

```
دانلود همه‌چیز ← ذخیره روی دیسک ← شروع پردازش
```

— وقتی ۲ ترابایت فضای خالی ندارید، وقتی خود دانلود روزها طول می‌کشد، یا وقتی واقعاً فقط چند هزار نمونه برای یک آزمایش سریع نیاز دارید، شکست می‌خورد.

**کاری که استریمینگ به‌جای آن انجام می‌دهد:**

```
دیتاست ← درخواست نمونه ← دانلود فقط همان نمونه ← پردازش ← نمونه‌ی بعدی
```

| | حالت استاندارد | حالت استریمینگ |
|---|---|---|
| الگوی دسترسی | `dataset[100]` — دسترسی تصادفی کار می‌کند | `for sample in dataset:` — **فقط ترتیبی** |
| مصرف دیسک | دانلود کامل از ابتدا | تقریباً صفر |
| مصرف RAM | بیشتر | کمتر |
| Shuffle | Shuffle کامل روی کل مجموعه | **فقط Shuffle بافری** — یک پنجره‌ی محدود در حافظه شافل می‌شود، نه کل دیتاست |
| زمان شروع آموزش | کند (منتظر دانلود کامل) | سریع |
| بهترین کاربرد | داده‌ی کوچک/متوسط، آزمایش تکراری، دسترسی مبتنی بر ایندکس | دیتاست‌های چند صد گیگابایتی/چند ترابایتی (Common Voice، GigaSpeech)، آموزش تک‌مرحله‌ای |

**چرا دسترسی تصادفی در استریمینگ از کار می‌افتد — دلیل مکانیکی آن:** کتابخانه هیچ ایندکسی به یک جریان (stream) که هنوز مصرف نشده ندارد. واقعاً نمی‌داند نمونه‌ی شماره‌ی ۱۰٬۰۰۰ از کجا شروع می‌شود، تا وقتی که تا آن نقطه را بخواند — دیتاست مثل یک iterator رفتار می‌کند، نه یک آرایه. این همان محدودیت بنیادینی است که با *هر* منبع داده‌ی مبتنی بر generator یا استریم‌شده از شبکه به آن برمی‌خورید؛ این محدودیت مخصوص صوت نیست، اما در صوت به‌ویژه بیشتر به چشم می‌آید چون فایل‌ها آن‌قدر بزرگ‌اند که استریمینگ اغلب انتخاب پیش‌فرض درست است.

**چه زمانی از استریمینگ استفاده کنیم:**
- ✅ دیتاست خیلی بزرگ است
- ✅ فضای دیسک محدود است
- ✅ فقط به یک عبور (pass) نیاز دارید (مثلاً یک اجرای آموزش)

**چه زمانی استفاده نکنیم:**
- ❌ دیتاست کوچک است
- ❌ به دسترسی تکراری و ایندکس‌شده به نمونه‌های خاص نیاز دارید

## واحد ۲ — نقشه‌ی وظایف هوش مصنوعی صوت

تقریباً هر محصول هوش مصنوعی صوت به یکی از هفت خانواده‌ی وظیفه تجزیه می‌شود. سؤال سازمان‌دهنده: **ورودی (تقریباً) همیشه صوت است — فقط دسته‌ی خروجی تغییر می‌کند.**

```
صوت
   │
   ├── تشخیص گفتار (ASR)              ← متن
   ├── دسته‌بندی صوت                    ← برچسب کلاس
   ├── تشخیص گوینده                    ← شناسه‌ی گوینده
   ├── تشخیص احساس گفتار                ← برچسب احساس
   ├── تبدیل متن به گفتار (TTS)          ← گفتار          [ورودی متن است، نه صوت]
   ├── تولید صوت                        ← موج صدای جدید   [ورودی یک Prompt است، نه صوت]
   └── ترجمه‌ی گفتار                    ← متن/گفتار به زبانی دیگر
```

### ۱. تشخیص خودکار گفتار (ASR)
**هدف:** تبدیل زبان گفتاری به متن.
```
🎤 «سلام به همه»  ←  📝 «سلام به همه»
```
مدل‌های پرچمدار: **Whisper**، **Wav2Vec2**. کاربردها: زیرنویس خودکار، دستیارهای صوتی، تایپ با صدا، پیاده‌سازی تماس‌های مرکز تماس.

### ۲. دسته‌بندی صوت
**هدف:** تشخیص اینکه یک صدا *چیست* — خروجی یک برچسب است، نه متن.
```
صوت ← «پارس سگ»  |  «صدای شلیک»  |  «موسیقی»  |  «باران»
```

### ۳. تشخیص گوینده
**هدف:** تشخیص اینکه *چه کسی* در حال صحبت است.
```
صوت ← «علی»  یا  «گوینده‌ی ناشناس»
```
کاربردها: احراز هویت مبتنی بر صدا، امنیت، شناسایی اینکه چه کسی در متن جلسات صحبت کرده است.

### ۴. تشخیص احساس گفتار
**هدف:** تشخیص حالت احساسی گوینده.
```
صوت ← خوشحال | عصبانی | غمگین
```
کاربرد گسترده در تحلیل مرکز تماس و نظارت بر کیفیت مکالمه.

### ۵. تبدیل متن به گفتار (TTS)
**هدف:** معکوس ASR — متن وارد می‌شود، گفتار خارج می‌شود.
```
«خوش آمدید» ← 🔊 (صوت گفتاری سنتزشده)
```
کاربردها: کتاب‌های صوتی، دوبله، دستیارهای صوتی، صداهای ناوبری GPS.

### ۶. تولید صوت
**هدف:** تولید صوت از یک Prompt — لازم نیست متنی *گفته* شود، و خروجی لازم نیست اصلاً گفتار باشد.
```
«موسیقی حماسی سینمایی» ← 🎵 (موسیقی تولیدشده)
«صدای امواج اقیانوس»    ← 🌊 (صدای محیطی تولیدشده)
```
مدل‌های قابل‌توجه: **MusicGen**، **AudioGen**، **Bark** (برای برخی کاربردها)، **Stable Audio**.

### ۷. ترجمه‌ی گفتار
**هدف:** ترکیب تشخیص و ترجمه — گفتار به یک زبان، به متن یا گفتار زبانی دیگر تبدیل می‌شود.
```
گفتار فارسی ← متن ← ترجمه ← گفتار انگلیسی
گفتار آلمانی ← متن انگلیسی
```
Whisper به‌طور بومی بخشی از این را پشتیبانی می‌کند؛ SeamlessM4T مخصوص همین کار ساخته شده است.

### جدول مقایسه‌ی مبتنی بر خروجی

| وظیفه | خروجی |
|---|---|
| ASR | متن |
| دسته‌بندی صوت | برچسب کلاس |
| تشخیص گوینده | هویت گوینده |
| تشخیص احساس | برچسب احساس |
| TTS | صوت (گفتار) |
| تولید صوت | صوت (موسیقی/جلوه‌ی صوتی/محیطی) |
| ترجمه‌ی گفتار | متن یا گفتار، به زبانی دیگر |

### تفاوت TTS و تولید صوت (یک نقطه‌ی رایج سردرگمی)

| | **تبدیل متن به گفتار (TTS)** | **تولید صوت** |
|---|---|---|
| ورودی | «سلام به همه» | «موسیقی حماسی سینمایی» |
| خروجی | صدای انسانی که **دقیقاً همان جمله** را می‌خواند | یک اثر صوتی کاملاً جدید — محتوا *تفسیر* می‌شود، نه عیناً خوانده می‌شود |
| محدودیت | محتوا توسط متن ورودی ثابت شده است | محتوا از یک Prompt تولید/تفسیر می‌شود |
| مقایسه با ASR | معکوس مستقیم ASR | معادلی در ASR ندارد — چیزی برای معکوس کردن وجود ندارد |

**چرا تولید سخت‌تر از تشخیص است — عدم‌تقارنی که همه‌جای یادگیری ماشین دیده می‌شود:** ASR فقط باید *حدس بزند* که از روی یک سیگنال ثابت و موجود چه گفته شده است. تولید صوت باید از هیچ، هر ویژگی آکوستیکی که صدا را باورپذیر می‌کند — محتوای فرکانسی، ریتم، پاکت دامنه، زمان‌بندی، هارمونیک‌ها، و پیوستگی لحظه‌به‌لحظه — را *سنتز* کند. در زمان استنتاج (inference) هیچ سیگنال حقیقت‌پایه‌ای برای تکیه کردن وجود ندارد، فقط یک Prompt. این عدم‌تقارن «تحلیل آسان‌تر از سنتز است» در کل یادگیری ماشین عمومیت دارد، اما در صوت به‌طرز غیرمعمولی نمایان است، چون انسان‌ها پیش‌داوری‌های ادراکی بسیار ظریفی برای «طبیعی بودن» صدا دارند — یک صدای مصنوعی با ۲٪ خطای زمان‌بندی اغلب فوراً قابل‌تشخیص است، به شکلی که یک خطای ۲٪ در، مثلاً، امتیاز اطمینان یک دسته‌بند تصویر، هرگز این‌طور نیست.

### اسکلت مشترک زیر هر هفت وظیفه

```
صوت ← پیش‌پردازش ← استخراج ویژگی ← شبکه‌ی عصبی ← پیش‌بینی
```

فقط **پیش‌بینی** تغییر می‌کند:

| وظیفه | «پیش‌بینی» چیست |
|---|---|
| Whisper (ASR) | متن |
| دسته‌بندی صوت | برچسب کلاس |
| تشخیص گوینده | شناسه‌ی گوینده |
| تشخیص احساس | برچسب احساس |

**نتیجه‌ی عملی:** چون این اسکلت مشترک است، ابزارها، کد پیش‌پردازش، و حتی بخش بزرگی از *شهود* آزادانه بین وظایف منتقل می‌شود. یک پایپلاین را خوب یاد بگیرید (ASR با Whisper انتخاب طبیعی اول است) و بقیه به‌جای موضوعات جدید، به «همان اسکلت، سر متفاوت» تبدیل می‌شوند.

### انتزاع `pipeline()` — اولین استنتاج واقعی شما

این واقعاً اولین نقطه در برنامه‌ی درسی است که یک مدل را اجرا می‌کنید، نه اینکه صرفاً داده را بررسی کنید.

**بدون pipeline، باید همه‌ی این‌ها را خودتان انجام دهید:**
```
بارگذاری صوت ← تغییر نرخ ← استخراج ویژگی ← بارگذاری مدل ← استنتاج ← رمزگشایی ← متن
```
این تقریباً ۳۰ تا ۵۰ خط کد است. `transformers.pipeline()` آن را به این خلاصه می‌کند:

```python
from transformers import pipeline
pipe = pipeline("automatic-speech-recognition", model="...")
result = pipe(audio)
# {"text": "اسم من علی است."}
```

**آنچه پشت پرده در هر فراخوانی اتفاق می‌افتد:**
```
فایل صوتی ← خواندن صوت ← تغییر نرخ (در صورت نیاز) ← Processor ← مدل (Whisper/Wav2Vec2) ← Decoder ← متن
```

این الگوی دقیق — `ورودی ← Pipeline ← خروجی`، که فقط *نوع* pipeline و *مدل* تغییر می‌کند — در سراسر هر وظیفه‌ی صوتی تکرار می‌شود:

| وظیفه | ورودی Pipeline | خروجی Pipeline |
|---|---|---|
| ASR | صوت | متن |
| دسته‌بندی | صوت | برچسب |
| TTS | متن | گفتار |
| ترجمه | گفتار | متن ترجمه‌شده |

**جایی که `pipeline()` دیگر ابزار مناسبی نیست:** برای نمونه‌سازی، یادگیری و آزمایش‌های سریع عالی است، اما سیستم‌های تولیدی معمولاً مستقیماً سراغ عناصر پایه‌ی `Processor` + `Model` می‌روند، برای کنترل دقیق‌تر روی batch کردن، جایگذاری GPU، و منطق رمزگشایی سفارشی. با `pipeline()` مثل یک داربست رفتار کنید که از آن فارغ‌التحصیل می‌شوید، نه یک وابستگی تولیدی.

```
فایل صوتی
     │
     ▼
  Pipeline
     ├── خواندن صوت
     ├── تغییر نرخ
     ├── Processor
     ├── مدل
     ├── Decoder
     ▼
   متن
```

- [ ] TTS دقیقاً متن را با صدای بلند می‌خواند؛ تولید صوت محتوای جدیدی از یک Prompt سنتز می‌کند — این دو را با هم اشتباه نگیرید.
- [ ] تولید ذاتاً سخت‌تر از تشخیص/تحلیل است — چیزی برای تکیه کردن در زمان استنتاج وجود ندارد.

## واحد ۳ — ترنسفورمرهای صوتی واقعاً چگونه کار می‌کنند

این واحد، سنگ‌بنای مفهومی کل این حوزه است. این واحد را درونی کنید و هر معماری بعدی — ASR، TTS، دسته‌بندی — به‌جای مطلب جدید، به شکل «این‌طوری همین بدنه را تطبیق می‌دهیم» خوانده می‌شود.

### چرا ترنسفورمرها جای RNN/LSTM/GRU را در صوت گرفتند

مدل‌های صوتی پیش از ترنسفورمر به‌صورت ترتیبی پردازش می‌کردند:

```
نمونه ۱ ← نمونه ۲ ← نمونه ۳ ← ...
```

این کار کند بود (بدون موازی‌سازی) و در وابستگی‌های دوربرد ضعیف بود (اطلاعات هنگام عبور از بازگشت (recurrence) افت می‌کند). در عوض، ترنسفورمرها **کل دنباله را همزمان** از طریق self-attention بررسی می‌کنند — دقیقاً همان استدلالی که RNN‌ها را در NLP کنار زد، بدون تغییر به اینجا منتقل شده است.

### اما صوت که متن نیست — پس یک معماری بومی-متن چگونه آن را مدیریت می‌کند؟

این سؤال واقعی است که این واحد پاسخ می‌دهد، و پایپلاین چهار مرحله‌ی مشخص دارد:

```
موج صدا ← Feature Extractor ← Embedding ← Encoder ترنسفورمر ← سر وظیفه ← پیش‌بینی
```

**مرحله‌ی ۱ — موج صدا.** اعداد اعشاری خام: `[0.01, 0.03, -0.2, ...]`. در این حالت برای ترنسفورمر بی‌معنی است — هنوز هیچ ساختاری وجود ندارد که یک لایه‌ی self-attention بتواند از آن بهره ببرد.

**مرحله‌ی ۲ — Feature Extractor.** مدل مستقیماً روی موج خام کار نمی‌کند.
```
موج صدا ← Feature Extractor ← بردار ویژگی
```
- **رویکرد کلاسیک:** ویژگی‌های دستی‌مهندسی‌شده — **MFCC** یا **Mel Spectrogram**.
- **رویکرد مدرن:** مدل‌هایی مثل Wav2Vec2 این مرحله را کاملاً با یک **CNN یادگرفته‌شده** جایگزین می‌کنند — مدل ویژگی‌های مفید خودش را کشف می‌کند به‌جای تکیه بر پردازش سیگنال دستی. این آینه‌ی همان روند گسترده‌تر یادگیری ماشین برای جایگزینی مهندسی ویژگی با نمایش‌های یادگرفته‌شده است، فقط این‌بار روی آکوستیک به‌جای پیکسل یا n-gram اعمال شده.

**مرحله‌ی ۳ — Embedding.** دقیقاً مثل NLP: هر فریم به یک بردار با ابعاد بالا تبدیل می‌شود.
```
فریم ← ۷۶۸ عدد
```

**مرحله‌ی ۴ — Encoder ترنسفورمر.**
```
Embedding ← Self-Attention ← Encoder ← نمایش زمینه‌ای (Contextual)
```
این از نظر ساختاری دقیقاً همان چیزی است که درون BERT اتفاق می‌افتد.

**Self-Attention چگونه به‌صورت ملموس کمک می‌کند:** در جمله‌ی «لطفاً پول را انتقال بده»، تشخیص درست «انتقال» ممکن است نیازمند توجه هم‌زمان به ابتدا و انتهای جمله باشد. Attention مستقیماً این وابستگی را یاد می‌گیرد. در صوت، دقیقاً همین مکانیزم به مدل اجازه می‌دهد به فریم‌هایی که صدها میلی‌ثانیه — یا چند ثانیه — قبل‌تر بودند توجه کند، که برای هر چیزی که معنایش در یک بازه‌ی آکوستیک طولانی پخش شده اهمیت زیادی دارد (مثلاً رفع ابهام یک کلمه بر اساس لحنی که چند ثانیه قبل تعیین شده).

**مرحله‌ی ۵ — سر وظیفه (Task Head).** این تنها مرحله‌ای است که بر اساس وظیفه تغییر می‌کند:

```
                    ┌── ASR             ← کاراکترها ← جمله
Encoder ترنسفورمر  ├── دسته‌بندی        ← برچسب
                    └── تشخیص گوینده    ← Embedding گوینده/کلاس
```

### پیش‌بین‌ترین انشعاب معماری: فقط-Encoder در برابر Encoder-Decoder

| | **Wav2Vec2** | **Whisper** |
|---|---|---|
| ساختار | `صوت ← Encoder ← پیش‌بینی` | `صوت ← Encoder ← Decoder ← متن` |
| دسته | **فقط-Encoder** | **Encoder + Decoder** |
| کاربرد معمول | ASR (از طریق CTC)، دسته‌بندی | ASR، ترجمه، تولید آزادانه‌تر و انعطاف‌پذیرتر |

- **کار Encoder:** فهمیدن ورودی. `صوت ← معنا`
- **کار Decoder:** تولید خروجی. `معنا ← متن`، گام‌به‌گام تولید می‌شود.

> **کلید طلایی برای خواندن هر مدل صوتی جدید:** پیش از خواندن هر چیز دیگری درباره‌ی یک مدل ناآشنا، بپرسید *«آیا این مدل Decoder دارد؟»* این یک واقعیت، بیشتر پروفایل قابلیت مدل را پیش‌بینی می‌کند — پیش‌بینی مستقیم به سبک CTC در برابر تولید انعطاف‌پذیر خودبازگشتی (autoregressive)، قابلیت ترجمه، سرعت نسبی، و کاربرد معمول.

### نمودار کامل معماری این واحد

```
صوت
  │
  ▼
موج صدا
  │
  ▼
استخراج ویژگی
  │
  ▼
Embedding‌ها
  │
  ▼
ترنسفورمر
  │
  ▼
نمایش (Representation)
  │
  ▼
سر وظیفه
  │
  ▼
پیش‌بینی
```

### چرا این معماری واقعاً برنده شد — سه ویژگی که فراتر از صوت تعمیم پیدا می‌کنند

- ✅ **پردازش موازی** — همه‌ی فریم‌ها همزمان پردازش می‌شوند، نه یکی‌یکی.
- ✅ **وابستگی‌های دوربرد** — Attention با فاصله افت نمی‌کند، آن‌طور که بازگشت (recurrence) عملاً می‌کند.
- ✅ **یادگیری انتقالی (Transfer Learning)** — یک‌بار روی میلیون‌ها ساعت صوت پیش‌آموزش دهید؛ سپس با هزینه‌ی کم برای هر وظیفه‌ی پایین‌دستی فاین‌تیون کنید. این *همان* استدلال اقتصادی است که کل پارادایم مدرن بر آن استوار است: بدنه‌ی مدل سرمایه‌ی قابل‌استفاده‌ی مجدد است، و فقط سر مدل یا یک فاین‌تیون سبک، هزینه‌ی نهایی مختص وظیفه است.

### نقشه‌ی صریح انتقال دانش از NLP

اگر از قبل BERT یا GPT را می‌شناسید، تقریباً ۸۰٪ مفاهیم این واحد از قبل برایتان آشناست. تنها تفاوت واقعی:

```
NLP:  متن     ← توکن‌ها              ← Embedding ← ترنسفورمر
صوت: موج صدا ← ویژگی‌ها/Embedding                ← ترنسفورمر
```

معماری پایه تقریباً یکسان است؛ آنچه تغییر می‌کند این است که *داده‌ی خام چگونه به بردارهایی تبدیل می‌شود که یک ترنسفورمر می‌تواند مصرف کند*.

### 📎 چک‌لیست حفظ‌کردنی — واحد ۳

۱. ترنسفورمر بدنه‌ی تقریباً هر مدل صوتی مدرن است.
۲. ورودی ترنسفورمر هرگز فایل WAV خام نیست — یک نمایش برداری (ویژگی/Embedding) از آن است.
۳. Self-attention به مدل اجازه می‌دهد بخش‌های دور از هم سیگنال صوتی را به یکدیگر مرتبط کند.
۴. بیشتر مدل‌ها یک بدنه‌ی ترنسفورمر مشترک دارند و فقط در **سر** مختص وظیفه (ASR، دسته‌بندی، تشخیص گوینده و غیره) تفاوت دارند.
۵. انشعاب کلیدی معماری: **Wav2Vec2 فقط-Encoder است؛ Whisper از نوع Encoder-Decoder است.** این یک واقعیت، بیشتر رفتار یک مدل را پیش‌بینی می‌کند.

## واحد ۴ — تشخیص خودکار گفتار (ASR)

### مسئله‌ی اصلی: هم‌ترازی (Alignment)

شما ۲ ثانیه صوت دارید ← ۳۲٬۰۰۰ نمونه. یک خروجی دارید: «سلام».

**کدام نمونه‌ها به کدام حروف مربوط می‌شوند؟** نمی‌دانید — و تقریباً هیچ دیتاست واقعی این سطح از جزئیات را ارائه نمی‌دهد. به نظر می‌رسد آموزش نیازمند این باشد:

```
۰٫۱۵ ثانیه ← س
۰٫۲۸ ثانیه ← ﻻ
۰٫۴۱ ثانیه ← م
```

اما دیتاست‌ها تقریباً هرگز این سطح از حاشیه‌نویسی زمانی را ندارند. تنها چیزی که واقعاً دارید این است:

```
صوت ← «سلام»
```

این «مسئله‌ی هم‌ترازی» است، و دلیل اینکه ASR «صرفاً» دسته‌بندی فریم‌به‌فریم نیست، همین است. دو پاسخ معماری وجود دارد که یک معامله‌ی واقعی و تکرارشونده در مدل‌سازی دنباله به‌طور کلی را نمایندگی می‌کنند.

### راه‌حل ۱: CTC (دسته‌بندی زمانی وابسته‌محور)

**ایده‌ی اصلی:** نیاز به هم‌ترازی زمانی در سطح فریم را کاملاً کنار بگذارید. مدل برای هر فریم *چیزی* پیش‌بینی می‌کند — از جمله یک **توکن خالی (blank)** ویژه به معنای «اینجا خروجی‌ای نیست» — و سپس دو قانون قطعی و ساده آن را به متن نهایی فشرده می‌کنند.

**خروجی خام و فریم‌به‌فریم مدل:**
```
فریم ۱ ← س
فریم ۲ ← س
فریم ۳ ← <خالی>
فریم ۴ ← ﻻ
فریم ۵ ← <خالی>
فریم ۶ ← م
فریم ۷ ← م
فریم ۸ ← <خالی>
فریم ۹ ← ...
```

**قانون رمزگشایی ۱ — فشرده کردن تکرارهای پیاپی:**
```
س س ← س
```

**قانون رمزگشایی ۲ — حذف همه‌ی توکن‌های خالی:**
```
س <خالی> ﻻ <خالی> م م <خالی> ... ← سلام
```

**چرا توکن خالی بینش واقعی است، نه یک جزئیات پیاده‌سازی:** بدون آن، مدل هیچ راهی برای تمایز بین «دو بار س پیش‌بینی کردم چون واقعاً دو صدای س جداگانه است» از «س را در سه فریم پیاپی پیش‌بینی کردم چون یک صدای س اتفاقاً سه فریم طول کشیده» ندارد. توکن خالی این ابهام را با علامت‌گذاری صریح «هنوز تصمیم نگرفته / الان کاراکتر جدیدی نیست» برطرف می‌کند، و همین است که قانون ۱ (فشرده کردن تکرارها) را ایمن می‌سازد.

**یک شهود ساده برای کل این مکانیزم:**
> تصور کنید به کسی می‌گویید: «هر وقت مطمئن نیستی، چیزی ننویس (خالی بگذار). هر وقت مطمئنی، همان حرف را بنویس — اگر همان حرف را چند بار پشت‌سرهم نوشتی، بعداً آن‌ها را با هم ادغام می‌کنیم.» این کل فلسفه‌ی CTC است. به همین دلیل CTC می‌تواند فقط از جفت‌های (صوت، متن پیاده‌شده) آموزش ببیند، بدون هیچ نیازی به برچسب‌های زمانی سطح کاراکتر.

**معماری:**
```
موج صدا ← Feature Extractor مبتنی بر CNN ← Encoder ترنسفورمر ← لایه‌ی خطی ← احتمالات کاراکتر ← Decoder ی CTC ← متن
```

**چرا Wav2Vec2 از CTC استفاده می‌کند:**
- ✅ نیازی به Decoder بزرگ نیست
- ✅ آموزش ساده‌تر
- ✅ سریع‌تر
- ✅ به‌طور خاص برای پیاده‌سازی گفتار مناسب است (برخلاف ترجمه یا بازنویسی)

**محدودیت جدی:** CTC فرض می‌کند ترتیب خروجی دقیقاً آینه‌ی ترتیب ورودی است (هم‌ترازی یکنوا/monotonic). این معماری **نمی‌تواند ترتیب را تغییر دهد**، آن‌طور که ترجمه نیاز دارد:
```
صوت ← «سلام دنیا»   (ترتیب خروجی باید با ترتیب ورودی مطابقت داشته باشد)
```
همین باعث می‌شود CTC برای ترجمه، خلاصه‌سازی، یا تولید آزاد متن نامناسب باشد — هر وظیفه‌ای که به تغییر ترتیب یا تبدیل محتوا نیاز داشته باشد، نه صرفاً پیاده‌سازی.

### راه‌حل ۲: Seq2Seq (Encoder-Decoder)

**ایده‌ی اصلی:** به‌جای یک خروجی برای هر فریم، کل ورودی را در یک نمایش معنایی فشرده کنید، سپس خروجی را به‌صورت خودبازگشتی — یک توکن در هر مرحله — *تولید* کنید، بدون هیچ الزامی که ترتیب خروجی آینه‌ی ترتیب ورودی باشد.

```
Encoder: صوت ← نمایش معنایی
Decoder: معنا ← س ← سﻻ ← سﻻم    (گام‌به‌گام تولید می‌شود)
```

**چرا Decoder اهمیت دارد:** در هر گام تولید، تصمیم می‌گیرد *«کدام کلمه باید بعدی باشد؟»* — و همین است که تغییر ترتیب را ممکن می‌کند:
```
ورودی:  "Je m'appelle Ali"
خروجی: «اسم من علی است»
```
مدل مجبور نیست ترتیب کلمات فرانسوی را حفظ کند تا خروجی فارسی درستی تولید کند.

**Attention درون Decoder:** در هر گام تولید، Decoder می‌پرسد *«کدام بخش از ورودی به توکنی که الان تولید می‌کنم مرتبط‌تر است؟»* — مثلاً، تولید کلمه‌ی «کامپیوتر» ممکن است توجه را به بخش خاصی از صوت ورودی جلب کند.

**معماری Seq2Seq ترنسفورمری (چیزی که Whisper واقعاً هست):**
```
صوت ← Audio Encoder ← نمایش پنهان (Hidden) ← Text Decoder ← متن تولیدشده
```
مثال: گفتار آلمانی وارد می‌شود، متن انگلیسی خارج می‌شود. یا گفتار فارسی وارد می‌شود، متن فارسی خارج می‌شود — Whisper هم پیاده‌سازی و هم ترجمه را از طریق همان معماری انجام می‌دهد، تنها با تفاوت در اینکه Decoder برای تولید چه چیزی prompt/آموزش داده شده است.

**یک شهود ساده برای این دو رویکرد، کنار هم:**

| | شهود |
|---|---|
| **CTC** | مثل کسی که همزمان با شنیدن صدا، حروف را حدس می‌زند: `س س ﻻ م ← سلام` |
| **Seq2Seq** | مثل یک مترجم انسانی: اول معنا را می‌فهمد (`صوت ← معنا`)، سپس جمله را تولید می‌کند (`معنا ← متن`) |

### CTC در برابر Seq2Seq — جدول کامل تصمیم‌گیری

| | **CTC** | **Seq2Seq (Encoder-Decoder)** |
|---|---|---|
| Decoder | ندارد (لایه‌ی خطی + رمزگشایی CTC) | دارد |
| تولید خروجی | مستقیم، فریم‌به‌فریم | گام‌به‌گام (خودبازگشتی) |
| مدیریت هم‌ترازی | با خود قانون فشرده‌سازی حل می‌شود | از طریق Attention یاد گرفته می‌شود |
| سرعت | سریع‌تر | کندتر |
| انعطاف‌پذیری | کمتر — ترتیب خروجی = ترتیب ورودی | بیشتر — می‌تواند ترتیب را تغییر دهد، ترجمه کند، بازنویسی کند |
| کیفیت ترجمه | ضعیف | عالی |
| مدل پرچمدار | Wav2Vec2 | Whisper |

**یک خلاصه‌ی یک‌خطی که ارزش به‌خاطر سپردن دارد:**
> CTC می‌گوید: «صدا را مستقیماً به حروف نگاشت کن.» Seq2Seq می‌گوید: «معنای صدا را بفهم، سپس متن درست را تولید کن.»

**چرا این حوزه به سمت Seq2Seq حرکت کرد:** کاربردهای واقعی فراتر از پیاده‌سازی ساده می‌روند — ترجمه‌ی همزمان، خلاصه‌سازی مکالمه، تولید پاسخ، رندر بین‌زبانی، و بهبود عمومی متن، همگی دقیقاً به همان انعطاف‌پذیری‌ای نیاز دارند که CTC از نظر ساختاری فاقد آن است.

### کل پایپلاین عملی ASR، مرحله‌به‌مرحله

```
صوت ← پیش‌پردازش ← استخراج ویژگی ← مدل گفتار ← Decoder ← متن
```

**۱. ورودی صوتی.** نمونه‌های خام `[0.02, 0.05, -0.01, ...]`، معمولاً پیش از رسیدن به مدل به یک نرخ نمونه‌برداری و طول یکسان (مثلاً ۱۶ کیلوهرتز) نرمال‌سازی می‌شوند.

**۲. استخراج ویژگی** — دو رویکرد رقیب:

| | دستی (کلاسیک) | یادگرفته‌شده (مدرن) |
|---|---|---|
| روش | MFCC — ویژگی‌های فرکانس، انرژی، تغییرات زمانی را استخراج می‌کند | یک رمزگذار ویژگی CNN، نمایش خودش را یاد می‌گیرد |
| پایپلاین | `صوت ← MFCC ← شبکه‌ی عصبی ← متن` | `صوت ← CNN Feature Encoder ← ترنسفورمر ← متن` |
| مثال | سیستم‌های قدیمی‌تر ASR | Wav2Vec2، Whisper |

**۳. خود مدل** — دو خانواده‌ای که اکنون می‌شناسید:

```
الف) مبتنی بر CTC (Wav2Vec2):
صوت ← Encoder ← احتمالات کاراکتر ← Decoder ی CTC ← متن

ب) Encoder-Decoder (Whisper):
صوت ← Encoder ← نمایش ← Decoder ← متن
   (Decoder گام‌به‌گام تولید می‌کند: <شروع> ← می ← می‌خواهم ← می‌خواهم ... ← می‌خواهم رزرو کنم)
```

**۴. رمزگشایی — تبدیل احتمالات به متن نهایی.** مدل مستقیماً متن تولید نمی‌کند؛ *احتمالات هر موقعیت* را تولید می‌کند:
```
س: ۰٫۸
ب: ۰٫۱
پ: ۰٫۰۵
```
Decoder باید بهترین مسیر کلی را در میان این احتمالات پیدا کند:

| استراتژی | نحوه‌ی کار | معامله |
|---|---|---|
| **رمزگشایی حریصانه (Greedy)** | در هر مرحله فقط بهترین گزینه انتخاب می‌شود | سریع، اما همیشه بهینه‌ی کلی نیست |
| **جستجوی پرتویی (Beam Search)** | چند مسیر کاندید را همزمان دنبال می‌کند (مثلاً «سلام»، «زلام»، «هلو»)، سپس بهترین کلی را انتخاب می‌کند | دقیق‌تر، اما از نظر محاسباتی سنگین‌تر |

**۵. ارزیابی** — برای شکست کامل معیارها به [واحد ۵](#واحد-۵--ارزیابی-asr-معیارهایی-که-واقعاً-اهمیت-دارند) مراجعه کنید؛ معیار اصلی **WER (نرخ خطای کلمه)** است.

### جایی که ASR واقعی در دنیای واقعی از کار می‌افتد (به‌ندرت «مدل بد است»)

| چالش | مثال ملموس |
|---|---|
| **نویز** | گفتار همراه با نویز پس‌زمینه |
| **لهجه** | تفاوت‌های تلفظ انگلیسی آمریکایی در برابر بریتانیایی در برابر هندی |
| **سرعت گفتار** | گفتار سریع و به‌هم‌پیوسته |
| **چند گوینده** | گویندگان هم‌پوشان یا به‌سرعت متناوب |
| **زبان‌های کم‌منبع** | فارسی و زبان‌های مشابهِ کم‌منبع از داده‌ی آموزشی محدود رنج می‌برند، که کل پایپلاین را دشوارتر می‌کند |

### 📎 چک‌لیست حفظ‌کردنی — واحد ۴

۱. ASR یعنی تبدیل گفتار به متن.
۲. دو معماری غالب مدرن: **CTC** (مثلاً Wav2Vec2) و **Encoder-Decoder** (مثلاً Whisper).
۳. مدل‌های مدرن به‌طور فزاینده نمایش صوتی خودشان را یاد می‌گیرند به‌جای تکیه بر ویژگی‌های دستی مثل MFCC.
۴. **Decoder** چیزی است که واقعاً متن نهایی خروجی را تولید یا انتخاب می‌کند.
۵. **WER (نرخ خطای کلمه)** معیار اصلی ارزیابی است.
۶. مدل تنها یک عامل است — کیفیت داده، نویز پس‌زمینه، لهجه، و در دسترس بودن منابع زبانی، همگی به‌طور جدی روی عملکرد ASR در دنیای واقعی اثر می‌گذارند.

> در پایان این واحد، عملاً بدنه‌ی سیستم‌هایی مثل Whisper، تبدیل گفتار به متن گوگل، Azure Speech، و بیشتر سیستم‌های تولیدی پیاده‌سازی تماس را فهمیده‌اید.

## واحد ۴ب — مدل‌های ASR از پیش‌آموزش‌دیده در عمل

**ایده‌ی اصلی:** به‌جای آموزش یک مدل تشخیص گفتار از صفر روی هزاران ساعت داده، از یک مدل از پیش‌آموزش‌دیده‌ی موجود استفاده کنید و فقط در صورت نیاز آن را روی داده‌ی خودتان فاین‌تیون کنید.

```
مدل از پیش‌آموزش‌دیده ← فاین‌تیون (در صورت نیاز) ← سیستم ASR سفارشی
```

### چرا مدل‌های از پیش‌آموزش‌دیده این‌قدر اهمیت دارند

آموزش ASR از صفر نیازمند این‌هاست: هزاران یا میلیون‌ها ساعت صوت، متن‌های پیاده‌شده‌ی متناظر، منابع GPU قابل‌توجه، و زمان آموزش طولانی. مدل‌هایی مثل Whisper از قبل بخش عظیمی از این هزینه را جذب کرده‌اند — استفاده‌ی مجدد از آن سرمایه‌گذاری تقریباً همیشه نقطه‌ی شروع درست است.

### خانواده‌های اصلی مدل‌های ASR/چندوظیفه‌ای

| مدل | معماری | ویژگی قابل‌توجه |
|---|---|---|
| **Wav2Vec2** | CNN ← Encoder ترنسفورمر ← **سر CTC** | ابتدا از طریق **یادگیری خودنظارتی (self-supervised)** روی صوت بدون برچسب پیش‌آموزش دیده، سپس روی یک مجموعه‌ی کوچک برچسب‌گذاری‌شده فاین‌تیون می‌شود — این جداسازی اهمیت زیادی دارد چون داده‌ی گفتاری برچسب‌گذاری‌شده گران است و خودنظارتی این نیاز را به‌شدت کاهش می‌دهد |
| **Whisper** (OpenAI) | Encoder-Decoder | روی داده‌ی عظیم آموزش دیده؛ هم ASR و هم ترجمه را به‌طور بومی مدیریت می‌کند؛ عملکرد چندزبانه‌ی قوی |
| **SpeechT5** | بدنه‌ی ترنسفورمر مشترک | چندوظیفه‌ای: ASR، TTS، و ترجمه‌ی گفتار از یک بدنه |
| **SeamlessM4T** (Meta) | Encoder-Decoder چندزبانه | گفتار↔متن و گفتار↔گفتار در بسیاری از زبان‌ها (مثلاً گفتار فارسی ← متن انگلیسی، یا متن انگلیسی ← گفتار فرانسوی) |

**بینش خودنظارتی Wav2Vec2، که ارزش تأمل دارد:** بیشتر پیشرفت این حوزه در چند سال اخیر از *جداسازی* «یاد گرفتن اینکه گفتار چگونه به نظر می‌رسد» (ارزان — صوت بدون برچسب فراوان است) از «یاد گرفتن نگاشت گفتار به این برچسب خاص» (گران — نیازمند متن پیاده‌شده) آمده است. Whisper شرط مخالفی بست: داده‌ی عظیم *ضعیف‌برچسب‌گذاری‌شده* به‌جای خودنظارتی رسمی. هر دو شرط جواب داد — که نشان می‌دهد یک دستور مقیاس‌دهی درست وجود ندارد، فقط معامله‌های متفاوتی بین هزینه‌ی داده، کیفیت برچسب، و بودجه‌ی محاسباتی.

### سه مؤلفه‌ای که واقعاً در Hugging Face با آن‌ها کار می‌کنید

```
صوت ← Processor ← Tensor ← مدل ← پیش‌بینی
```

| مؤلفه | نقش |
|---|---|
| **Pipeline** | ساده‌ترین رابط — `pipeline("automatic-speech-recognition", model="...")` |
| **Processor** | رابط بین صوت خام و مدل: تغییر نرخ، تبدیل موج به Tensor، آماده‌سازی ورودی |
| **Model** | خود شبکه‌ی عصبی: `Tensor ← شبکه‌ی عصبی ← پیش‌بینی` |

### فاین‌تیون، به‌شکل یک قیاس ساده

```
مدل از پیش‌آموزش‌دیده + دیتاست شما ← مدل فاین‌تیون‌شده
```

چرا این روش بهتر از آموزش از صفر است: مدل از قبل ساختار زبان، الگوهای آکوستیک، و تلفظ را یاد گرفته — فاین‌تیون فقط باید آن را تخصصی کند.

> **قیاسی که منبع اصلی استفاده می‌کند، و قیاس خوبی است:** فاین‌تیون کردن یک مدل بزرگ از پیش‌آموزش‌دیده، مثل تربیت کردن یک پزشک عمومی به یک متخصص است، نه آموزش دادن یک پزشک جدید از صفر.

### انتخاب مدل مناسب برای کار

| نیاز | انتخاب پیشنهادی |
|---|---|
| ASR سریع و سبک | Wav2Vec2 |
| عملکرد چندزبانه‌ی قوی | Whisper |
| ترجمه‌ی گفتار | Whisper / SeamlessM4T |
| سیستم چندوظیفه‌ای | SpeechT5 |

**معامله‌ی «بزرگ‌تر همیشه بهتر نیست»، به‌صورت ملموس:**

| | Whisper Large | Whisper Small/Base |
|---|---|---|
| دقت | ✅ عالی | ❌ پایین‌تر |
| نیاز به منابع | ❌ بالا | ✅ پایین |
| سرعت | کندتر | ✅ سریع‌تر |
| استقرار روی Edge | تناسب ضعیف | ✅ کاملاً مناسب |

برای سیستم‌های آفلاین یا با محدودیت منابع، همیشه بین **دقت در برابر تأخیر در برابر حافظه** در حال متعادل‌سازی هستید — انتخاب جهانی درستی وجود ندارد، فقط انتخاب درست برای محدودیت‌های استقرار شما.

### 📎 چک‌لیست حفظ‌کردنی — واحد ۴ب

۱. مدل‌های ASR از پیش‌آموزش‌دیده از قبل روی حجم عظیمی از داده آموزش دیده‌اند و آماده‌ی استفاده‌ی مستقیم هستند.
۲. مدل‌های کلیدی: **Wav2Vec2** (Encoder + CTC، سبک‌تر، متمرکز بر ASR) · **Whisper** (Encoder-Decoder، چندزبانه‌ی قوی) · **SpeechT5 / SeamlessM4T** (بیشتر چندوظیفه‌ای‌محور).
۳. در Hugging Face معمولاً با سه مؤلفه کار می‌کنید: **Pipeline** (ساده‌ترین) ← **Processor** (آماده‌سازی ورودی) ← **Model** (خود شبکه).
۴. فاین‌تیون = تطبیق یک مدل آماده با دیتاست خاص شما، نه ساختن از صفر.
۵. در پروژه‌های واقعی، انتخاب مدل فقط دقت نیست — سرعت، حافظه، سخت‌افزار، و کاربرد، همه اهمیت دارند.

> **چارچوبی مهندسی که ارزش درونی کردن دارد:** در پروژه‌های واقعی، بهترین استراتژی معمولاً «بزرگ‌ترین مدل را انتخاب کن» نیست. **مدل از پیش‌آموزش‌دیده + داده‌ی واقعیِ حوزه‌ی هدف + فاین‌تیون هوشمندانه = یک سیستم ASR واقعاً کاربردی.**

## واحد ۴ج — انتخاب و ساخت دیتاست‌های ASR

**سؤال اصلی:** برای آموزش یا ارزیابی یک مدل گفتار-به-متن از کدام دیتاست استفاده کنید؟ بسیاری فرض می‌کنند مدل مهم‌ترین متغیر است — در ASR، **کیفیت و ترکیب داده معمولاً اهمیت بیشتری دارد.**

### چرا انتخاب دیتاست به‌طرز نامتناسبی مهم است

یک مدل ASR فقط چیزی را یاد می‌گیرد که در داده‌ی آموزشی دیده است:

| اگر دیتاست فقط شامل... باشد | آنگاه مدل... |
|---|---|
| صدای گویندگان جوان | با گویندگان مسن‌تر مشکل خواهد داشت |
| صوت با کیفیت استودیویی | در محیط‌های پرنویز از کار خواهد افتاد |
| یک لهجه | با لهجه‌های دیگر مشکل خواهد داشت |

> **چارچوبی که ارزش درونی کردن دارد:** *دیتاست همان تجربه‌ای است که مدل از دنیای واقعی به دست می‌آورد.* هر شکافی در دیتاست، به یک شکاف در عملکرد استقرارشده تبدیل می‌شود — و برخلاف انتخاب معماری مدل، شکاف داده تا زمان شکست در محیط تولید نامرئی می‌ماند.

### ساختار یک دیتاست ASR

هر نمونه به دو چیز نیاز دارد:
```
صوت + متن پیاده‌شده
```
مثلاً `audio_001.wav` جفت‌شده با «نزدیک‌ترین بانک کجاست؟» — که به مدل یاد می‌دهد `این صدا ← این متن`.

### چهار ویژگی‌ای که کیفیت دیتاست را تعیین می‌کنند

**۱. کیفیت صوت.** نرخ نمونه‌برداری، سطح نویز، کیفیت میکروفون، فرمت فایل را بررسی کنید.

| دیتاست حرفه‌ای | شرایط واقعی استقرار |
|---|---|
| ۱۶ کیلوهرتز | ضبط تلفنی |
| گفتار تمیز | نویز پس‌زمینه |
| تک‌گوینده | چند گوینده |

**۲. کیفیت متن پیاده‌شده.** متن جفت‌شده باید واقعاً دقیق باشد — و «دقیق» خودش یک قضاوت است:
```
صوت: «می‌خوام برم خونه»
گزینه‌ی متن الف: «می‌خواهم بروم خانه»   (نرمال‌شده)
گزینه‌ی متن ب: «می‌خوام برم خونه»       (عیناً همان‌طور که گفته شده)
```
اینکه کدام درست است به هدف شما بستگی دارد — برای ASR عمومی، فرم عیناً گفته‌شده معمولاً هدف آموزشی بهتری است.

**۳. تنوع.** یک دیتاست خوب این‌ها را پوشش می‌دهد:
- **گویندگان** — مرد، زن، سنین متنوع
- **لهجه‌ها** — لهجه‌های منطقه‌ای/ملی متعدد
- **شرایط ضبط** — اتاق ساکت، نویز خیابان، تماس‌های تلفنی

**۴. حجم — اما کیفیت بر کمیت غالب است.** داده‌ی بیشتر عموماً بهتر است، اما:
> **۱٬۰۰۰ ساعت صوت با کیفیت پایین می‌تواند بدتر از ۱۰۰ ساعت صوت با کیفیت بسیار بالا باشد.**

### دیتاست‌های شناخته‌شده‌ی ASR

| دیتاست | سازنده/منشأ | ویژگی‌های کلیدی | کاربرد معمول |
|---|---|---|---|
| **Common Voice** | Mozilla | چندزبانه، حجم بالا، پوشش گسترده‌ی زبانی | آموزش میان‌زبانی |
| **LibriSpeech** | — | حدود ۱٬۰۰۰ ساعت، برگرفته از کتاب صوتی، کیفیت نسبتاً بالا | محک‌زنی استاندارد |
| **TED-LIUM** | سخنرانی‌های TED | گفتار طبیعی‌تر، جملات طولانی‌تر | ارزیابی گفتار طبیعی |
| **GigaSpeech** | — | مقیاس بسیار بزرگ — پادکست‌ها، گفتار دنیای واقعی | آموزش مدل‌های بزرگ |

### تطبیق دیتاست فاین‌تیون شما با محیط استقرار

اگر یک مدل عمومی (مثلاً Whisper) را برای یک حوزه‌ی خاص (مثلاً تماس‌های پشتیبانی مشتریان فارسی‌زبان) سفارشی‌سازی می‌کنید، داده‌ی فاین‌تیون شما باید شبیه **شرایط واقعی استقرار** باشد، نه یک تقریب تمیز استودیویی از آن:

```
❌ بد:   گفتار استودیویی ← مدل تماس‌های مشتری
✅ خوب: تماس‌های واقعی مشتریان ← مدل تماس‌های مشتری
```

### آموزش / اعتبارسنجی / آزمون — و تله‌ی نشت مختص صوت

تقسیم استاندارد:
```
دیتاست
  ├── آموزش (Train)         (۸۰٪)  — مدل یاد می‌گیرد
  ├── اعتبارسنجی (Validation) (۱۰٪)  — تنظیم هایپرپارامتر/انتخاب مدل
  └── آزمون (Test)           (۱۰٪)  — ارزیابی نهایی و صادقانه
```

**نشت داده — تله‌ای واقعاً مختص صوت:** یک گوینده‌ی یکسان نباید هم در بخش آموزش و هم در بخش آزمون ظاهر شود.
```
❌ آموزش: صدای علی     آزمون: صدای علی
```
اگر این اتفاق بیفتد، ممکن است مدل به‌سادگی *صدای علی* را حفظ کند به‌جای اینکه ASR عمومی یاد بگیرد — دقت آزمون شما عالی به نظر خواهد رسید ولی بی‌معنا خواهد بود. این ظریف‌تر از ریسک نشت مشابه در NLP است، چون هویت گوینده سیگنالی قوی و به‌سادگی قابل‌بیش‌برازش (overfit) است که هیچ ربطی به خود وظیفه‌ی پیاده‌سازی ندارد.

### برای زبان‌های کم‌منبع (مثلاً فارسی)

چالش از نظر ساختاری بزرگ‌تر است. راهکارهای رایج کاهش این چالش:
- استفاده از دیتاست‌های چندزبانه
- فاین‌تیون کردن مدل‌های بزرگ از پیش‌آموزش‌دیده به‌جای آموزش از صفر
- **افزایش داده (Data Augmentation)** — گسترش مصنوعی آنچه دارید:
  - افزودن نویز
  - تغییر سرعت
  - تغییر پرده‌ی صدا (pitch)

### 📎 چک‌لیست حفظ‌کردنی — واحد ۴ج

۱. یک دیتاست ASR اساساً جفت‌های `صوت + متن پیاده‌شده` است.
۲. کیفیت دیتاست مستقیماً کیفیت مدل را تعیین می‌کند — احتمالاً بیشتر از انتخاب معماری.
۳. تنوع گوینده، تنوع لهجه، و تنوع شرایط ضبط، همگی حیاتی هستند.
۴. دیتاست باید شبیه محیط واقعی استقرار مدل باشد، نه یک نسخه‌ی ایده‌آل‌شده‌ی استودیویی از آن.
۵. داده باید به آموزش/اعتبارسنجی/آزمون تقسیم شود — و هویت گوینده یک بردار نشت کلاسیک و به‌راحتی قابل‌اغماض، مختص صوت، است.
۶. برای زبان‌های کم‌منبع، **مدل از پیش‌آموزش‌دیده + فاین‌تیون** معمولاً عملی‌ترین مسیر است.

## واحد ۴د — بررسی عمیق معماری دسته‌بندی صوت

پس از پوشش دو معماری برای تبدیل گفتار به متن (CTC، Seq2Seq)، این واحد به یک سؤال اساساً متفاوت می‌پردازد:
> *«این صدا چیست؟»* — نه *«چه گفته شد؟»*

### دسته‌بندی صوت چیست

**هدف:** گرفتن یک فایل صوتی و اختصاص یک برچسب به آن.
```
🔊 audio.wav ← «پارس سگ» | «گفتار» | «موسیقی» | «صدای موتور»
```

| | ASR | دسته‌بندی صوت |
|---|---|---|
| مثال | `صوت ← «ساعت چند است؟»` | `صوت ← «گفتار»` |
| چیزی که اهمیت دارد | **محتوای** آنچه گفته شده | **نوع/دسته‌ی** خود صدا |

### معماری استاندارد

```
موج صدا ← Feature Extractor ← Audio Encoder ← سر دسته‌بندی ← برچسب کلاس
```

**مرحله‌ی ۱ — تبدیل ویژگی.** بیشتر معماری‌ها نمی‌توانند مستقیماً روی موج خام کار کنند:
```
موج صدا ← Mel Spectrogram   (یا)   موج صدا ← ویژگی‌های یادگرفته‌شده
```

**مرحله‌ی ۲ — Encoder.** مدل ویژگی‌های آکوستیک مرتبط با وظیفه را یاد می‌گیرد. برای تشخیص پارس سگ: امضای فرکانسی پارس، الگوی زمانی، شدت صدا. برای تشخیص موسیقی: ریتم، هارمونی، محتوای فرکانسی.

**مرحله‌ی ۳ — سر دسته‌بندی.** خروجی Encoder یک بردار است، مثلاً `[0.23, 0.71, 0.05, ...]`، که از این مسیر عبور می‌کند:
```
لایه‌ی خطی ← Softmax ← احتمال
```
خروجی نمونه: «گفتار: ۰٫۹۵، موسیقی: ۰٫۰۳، نویز: ۰٫۰۲» ← پیش‌بینی نهایی: **گفتار**.

### دسته‌بندی مبتنی بر ترنسفورمر، و چرا Pooling لازم است

```
صوت ← Feature Extractor ← Encoder ترنسفورمر ← Pooling ← دسته‌بند ← برچسب
```

**چرا Pooling لازم است:** فایل‌های صوتی طول متغیری دارند (۳ ثانیه در برابر ۲۰ ثانیه)، اما دسته‌بندی به یک پاسخ با اندازه‌ی ثابت برای هر فایل نیاز دارد. Pooling (مثلاً **میانگین‌گیری/Average Pooling**) دنباله‌ی فریم‌های با طول متغیر را در یک بردار با طول ثابت فشرده می‌کند:
```
فریم‌ها ← میانگین‌گیری ← یک بردار
```

### مدل‌های قابل‌توجه برای این وظیفه

**Wav2Vec2 برای دسته‌بندی** — گرچه بیشتر برای ASR شناخته می‌شود، Encoder آن قابل‌استفاده‌ی مجدد است:
```
صوت ← Encoder ی Wav2Vec2 ← دسته‌بند ← احساس / گوینده / رویداد
```

**Audio Spectrogram Transformer (AST)** — از نظر مفهومی از Vision Transformer وام گرفته شده: به‌جای پچ‌های تصویر، روی پچ‌های اسپکتروگرام عمل می‌کند:
```
صوت ← اسپکتروگرام ← پچ‌ها ← ترنسفورمر ← کلاس
```

### مثال‌های کاربردی ملموس

| کاربرد | ورودی | نمونه‌های خروجی |
|---|---|---|
| تشخیص احساس | صدای فرد | عصبانی، خوشحال، غمگین، خنثی |
| تشخیص رویداد صوتی | صدای محیطی | باران، بوق ماشین، پارس سگ، آژیر |
| تشخیص زبان | گفتار | فارسی، انگلیسی، آلمانی |
| تشخیص گوینده | نمونه‌ی صدا | گوینده‌ی الف |

### نقشه‌ی وظیفه به معماری، به‌روزشده

| وظیفه | معماری معمول |
|---|---|
| ASR | CTC یا Seq2Seq |
| دسته‌بندی صوت | Encoder + سر دسته‌بند |
| TTS | Seq2Seq / تولیدی |
| تولید صوت | مدل‌های تولیدی |

### الگوی مهم معماری: یک Encoder مشترک، چند سر

در عمل، بسیاری از سیستم‌های صوتی مدرن **یک Encoder صوتی عمومی** می‌سازند و فقط سر را عوض می‌کنند:

```
              Encoder صوتی مشترک
                        │
        ┌───────────────┼───────────────┐
        ▼               ▼               ▼
      ASR            احساس         تشخیص گوینده
      سر               سر               سر
```

بخش سخت — واقعاً فهمیدن صوت خام — یک‌بار یاد گرفته می‌شود و در بسیاری از کاربردهای پایین‌دستی مجدداً استفاده می‌شود. این همان الگوی «مدل پایه + سرهای سبک» است که از بینایی ماشین و زبان می‌شناسید، اینجا روی نمایش‌های آکوستیک اعمال شده.

### 📎 چک‌لیست حفظ‌کردنی — واحد ۴د

۱. دسته‌بندی صوت **نوع/دسته‌ی** صدا را تشخیص می‌دهد، نه محتوای گفته‌شده.
۲. معماری استاندارد: `صوت ← Feature Extractor ← Encoder ← دسته‌بند ← برچسب`.
۳. خروجی Encoder یک نمایش معنایی از صوت است.
۴. چون طول فایل‌ها متغیر است، **Pooling** دنباله‌های با طول متغیر را در یک بردار ثابت فشرده می‌کند.
۵. Wav2Vec2 و Audio Spectrogram Transformer (AST) هر دو به‌طور رایج در اینجا استفاده می‌شوند.
۶. تمایز اصلی از ASR: ASR می‌پرسد *«چه گفته شد؟»*؛ دسته‌بندی می‌پرسد *«این چه نوع صدایی است؟»*

## واحد ۵ — ارزیابی ASR: معیارهایی که واقعاً اهمیت دارند

### نرخ خطای کلمه (WER) — معیار اصلی

```
WER = (S + D + I) / N
```
- **S** (جایگزینی/Substitution) — یک کلمه به‌اشتباه عوض شده: `قهوه ← کپی`
- **D** (حذف/Deletion) — یک کلمه کاملاً حذف شده: «من واقعاً قهوه دوست دارم» ← «من قهوه دوست دارم»
- **I** (افزایش/Insertion) — یک کلمه‌ی اضافه افزوده شده: «من قهوه دوست دارم» ← «من واقعاً قهوه دوست دارم»
- **N** — تعداد کل کلمات در متن مرجع (حقیقت پایه)

**مثال حل‌شده:**
```
مرجع:  من یادگیری ماشین را دوست دارم
فرضیه: من یادگیری ماشینی را دوست دارم
```
```
من        ✓
یادگیری   ✓
ماشین     ✗   ← جایگزینی: ماشین ← ماشینی
را        ✓
دوست دارم ✓
```
یک جایگزینی از ۵ کلمه‌ی مرجع ← **WER = ۱/۵ = ۲۰٪**. WER پایین‌تر یعنی بهتر.

### سه نوع خطای WER، هرکدام با یک مثال حداقلی

| نوع | مرجع | فرضیه | چه اتفاقی افتاد |
|---|---|---|---|
| **جایگزینی (S)** | «من قهوه دوست دارم» | «من کپی دوست دارم» | `قهوه` ← `کپی` |
| **حذف (D)** | «من واقعاً قهوه دوست دارم» | «من قهوه دوست دارم» | `واقعاً` حذف شده |
| **افزایش (I)** | «من قهوه دوست دارم» | «من واقعاً قهوه دوست دارم» | `واقعاً` افزوده شده |

**چرا WER استاندارد است:** کل هدف ASR تولید متن درست است — رفتار درونی مدل بی‌ربط است اگر تعداد کلمات تحویل‌شده اشتباه باشد. WER دقیقاً همان سؤالی را پاسخ می‌دهد که مهم است: *چند کلمه را درست تشخیص داد؟*

### نرخ خطای کاراکتر (CER)

همان فرمول، اما اعمال‌شده در سطح **کاراکتر** به‌جای سطح کلمه:
```
مرجع:  hello
فرضیه: helo      ← یک کاراکتر حذف شده
```

| معیار | واحد | مناسب برای |
|---|---|---|
| **WER** | کلمه | انگلیسی، ASR مکالمه‌ای عمومی |
| **CER** | کاراکتر | زبان‌های با ساختار واژگانی پیچیده (مثلاً چینی)، برخی کاربردهای فارسی |

### فاکتور زمان-واقعی (RTF) — سرعت، نه فقط دقت

```
RTF = زمان پردازش / مدت زمان صوت
```
**مثال:** یک فایل صوتی ۱ دقیقه‌ای که در ۱۰ ثانیه پردازش می‌شود:
```
RTF = ۱۰ / ۶۰ = ۰٫۱۶
```

| مقدار RTF | معنا |
|---|---|
| `< 1` | سریع‌تر از زمان واقعی |
| `= 1` | دقیقاً زمان واقعی |
| `> 1` | کندتر از زمان واقعی |

### تأخیر (Latency) — حیاتی برای سیستم‌های تعاملی

**تعریف:** چقدر طول می‌کشد تا بعد از صحبت کردن، سیستم نتیجه تولید کند؟ برای یک دستیار صوتی، فاصله‌ی بین گفتن «سلام» توسط کاربر و دریافت پاسخ ممکن است ۵۰ میلی‌ثانیه یا ۵ ثانیه باشد — و این فاصله اغلب از نظر ادراکی مهم‌تر از دقت خام است.

### معیارهای استحکام (Robustness) — تنها WER کافی نیست

در دنیای واقعی، باید بدانید مدل در برابر این‌ها چگونه عمل می‌کند:
- **نویز** — گفتار همراه با نویز پس‌زمینه
- **لهجه** — لهجه‌های منطقه‌ای متفاوت
- **گوینده** — صداهای متفاوت، سنین متفاوت، جنسیت‌های متفاوت

### دیتاست‌های محک — برای مقایسه‌ی منصفانه

مدل‌ها روی دیتاست‌های استانداردشده (LibriSpeech، Common Voice، TED-LIUM) مقایسه می‌شوند، دقیقاً به همین دلیل که مقایسه‌ها سیب با سیب باشند.

### معامله‌ای که جدول‌های امتیاز فقط-WER پنهان می‌کنند

**یک مقایسه‌ی نمایشی ملموس:**

| | مدل الف | مدل ب |
|---|---|---|
| WER | ۵٪ | ۸٪ |
| تأخیر | ۱۰ ثانیه | ۱۰۰ میلی‌ثانیه |

برای یک دستیار صوتی موبایل، **مدل ب کاملاً محتمل است انتخاب بهتری باشد** — با وجود WER «بدتر» — چون در آن زمینه، تأخیر تعاملی بیشتر از چند درصد دقت اضافه اهمیت دارد.

> **فرمول عملی:** یک ارزیابی واقعاً خوب برای ASR دنیای واقعی، **دقت + سرعت + حافظه + تأخیر** را با هم ترکیب می‌کند — هرگز فقط دقت. جدول‌های امتیاز محک برای بُعد اول بهینه می‌شوند و سه بُعد دیگر را به‌طور خاموش پنهان می‌کنند.

### مدل ذهنی کامل ارزیابی

```
مدل ASR
   │
   ▼
پیش‌بینی ← متن تولیدشده
   │
   ▼
مقایسه با مرجع
   │
   ├── WER
   ├── CER
   ├── تأخیر
   ├── RTF
   └── استحکام
```

### 📎 چک‌لیست حفظ‌کردنی — واحد ۵

۱. معیار اصلی ASR **WER (نرخ خطای کلمه)** است.
۲. WER سه نوع خطا را حساب می‌کند: **جایگزینی، حذف، افزایش**.
۳. **CER** خطای سطح کاراکتر را اندازه می‌گیرد — مفید برای زبان‌های با ساختار واژگانی پیچیده.
۴. در سیستم‌های تولیدی، سرعت و تأخیر به اندازه‌ی دقت خام اهمیت دارند.
۵. یک ارزیابی عملی و کامل ASR شامل: **دقت (WER/CER) + سرعت (RTF) + تجربه‌ی کاربری (تأخیر) + استحکام** است.

> **پیوند به معماری:** مدل‌های مبتنی بر CTC (Wav2Vec2) و مدل‌های Encoder-Decoder (Whisper) هر دو معمولاً با WER روی خروجی رمزگشایی‌شده ارزیابی می‌شوند، سپس در چند دیتاست محک مقایسه می‌شوند. یک مدل ASR واقعاً عالی صرفاً مدلی نیست که متن درست تولید کند — باید در محدودیت‌های زمانی و سخت‌افزاری هدف واقعی استقرارش نیز کار کند.

## واحد ۵ب — فاین‌تیون ASR در عمل

**سؤال واقعی که این واحد پاسخ می‌دهد:** مدل‌های از پیش‌آموزش‌دیده مثل Whisper و Wav2Vec2 وجود دارند و می‌توان مستقیماً از آن‌ها استفاده کرد — اما اگر مدل عمومی به‌سادگی برای کاربرد خاص شما به‌اندازه‌ی کافی خوب نباشد چه؟

### فاین‌تیون واقعاً چیست

> بازآموزش یک مدل از قبل پیش‌آموزش‌دیده روی داده‌ی اختصاصی خودتان، تا در یک وظیفه‌ی خاص بهبود یابد.

```
ASR از پیش‌آموزش‌دیده (مثلاً Whisper انگلیسی عمومی)
        +
دیتاست تماس بانکی (مثلاً تماس‌های پشتیبانی بانکی فارسی)
        ↓
ASR فاین‌تیون‌شده
```

### چرا فاین‌تیون بر آموزش از صفر برتری دارد

| | آموزش از صفر | فاین‌تیون |
|---|---|---|
| نقطه‌ی شروع | وزن‌های تصادفی | مدلی که از قبل الگوهای آکوستیک، ساختار زبان، تلفظ را می‌فهمد |
| داده‌ی موردنیاز | عظیم | نسبتاً کم، اختصاصیِ همان حوزه |
| محاسبات موردنیاز | بسیار بالا | بسیار کمتر |
| زمان موردنیاز | طولانی | کوتاه |

فاین‌تیون فقط باید دانش موجود مدل را با حوزه‌ی جدید شما **هم‌راستا** کند — نه اینکه آن دانش را از صفر بسازد.

### پایپلاین عمومی فاین‌تیون

```
دیتاست ← پیش‌پردازش ← استخراج ویژگی ← مدل از پیش‌آموزش‌دیده ← آموزش ← ارزیابی ← مدل فاین‌تیون‌شده
```

**۱. آماده‌سازی دیتاست** — جفت‌های `صوت + متن پیاده‌شده`، مثلاً `call_001.wav` + «شماره حساب من را بررسی کنید».

**۲. آماده‌سازی صوت** — تغییر نرخ (مثلاً `۴۸۰۰۰ هرتز ← ۱۶۰۰۰ هرتز`)، نرمال‌سازی صدا، حذف نمونه‌های خراب.

**۳. آماده‌سازی متن** — توکن‌سازی به شناسه‌هایی که مدل می‌تواند مصرف کند: «سلام» ← `[15, 32, 44, ...]`.

**۴. آموزش** — در هر گام: مدل صوت را دریافت می‌کند، یک پیش‌بینی تولید می‌کند (مثلاً «سﻻم»)، این با حقیقت پایه («سلام») مقایسه می‌شود، یک **loss** محاسبه می‌شود، و وزن‌ها کمی تنظیم می‌شوند.

**۵. ارزیابی** — بررسی WER قبل و بعد:
```
قبل از فاین‌تیون: WER = ۲۰٪
بعد از فاین‌تیون: WER = ۸٪
```

### نکات فاین‌تیون مختص هر معماری

**فاین‌تیون کردن Wav2Vec2:**
```
صوت ← Encoder ی Wav2Vec2 ← سر CTC ← متن
```
معمولاً، وزن‌های Encoder مستقیماً تنظیم می‌شوند.

**فاین‌تیون کردن Whisper:**
```
صوت ← Encoder ← Decoder ← متن
```
چون مدل بزرگ است، رایج است بخش‌هایی از آن را **منجمد (freeze)** کرده و فقط بخش‌های خاصی را آموزش دهید.

### دو استراتژی کاهش هزینه برای مدل‌های بزرگ

**۱. منجمد کردن لایه (Layer Freezing)**
```
Encoder ❄️  منجمد
Decoder 🔥  قابل‌آموزش
```
فایده: ردپای حافظه‌ی کمتر، آموزش سریع‌تر.

**۲. فاین‌تیون بهره‌ور از نظر پارامتر (PEFT)، مثلاً LoRA**
به‌جای به‌روزرسانی میلیاردها پارامتر، فقط تعداد کمی پارامتر تازه‌افزوده‌شده آموزش می‌بینند. این همان راهبرد کلی PEFT است — از نظر معماری مختص صوت نیست — که اینجا روی یک بدنه‌ی گفتاری اعمال شده.

### فاین‌تیون واقعاً چه زمانی ارزشش را دارد؟

**احتمالاً از آن صرف‌نظر کنید اگر:**
- زبان هدف رایج/عمومی است
- صوت معمولی/استاندارد است
- مدل پایه از قبل قوی است

**احتمالاً این کار را انجام دهید اگر:**
- اصطلاحات تخصصی حوزه دارید
- لهجه‌ی خاصی برای مدیریت دارید
- محیط پرنویز است
- زبان هدف کم‌منبع است

### حالت شکست حیاتی: بیش‌برازش (Overfitting)

اگر دیتاست فاین‌تیون کوچک باشد، مدل ممکن است به‌سادگی همان نمونه‌های دقیق را **حفظ کند** به‌جای تعمیم‌دهی:
```
آموزش‌دیده روی: ۱۰۰ نفر از یک شرکت
سپس آزموده‌شده روی: افراد جدید
← افت شدید عملکرد
```

**راهکارهای کاهش این ریسک:**
- داده‌ی بیشتر
- افزایش داده
- روش‌شناسی اعتبارسنجی مناسب

### افزایش داده برای ASR — تولید تنوع مصنوعی

با شروع از صوت واقعی (مثلاً «سلام»)، نمونه‌های متنوع را با این روش‌ها تولید کنید:
- افزودن نویز
- تغییر سرعت
- تغییر پرده‌ی صدا
- شبیه‌سازی کیفیت صوتی تماس تلفنی

این کار به‌طور مصنوعی تنوع مؤثر یک دیتاست کوچک را بدون نیاز به ضبط‌های واقعی جدید گسترش می‌دهد.

### 📎 چک‌لیست حفظ‌کردنی — واحد ۵ب

۱. فاین‌تیون یک مدل ASR آماده را با داده‌ی اختصاصی شما تطبیق می‌دهد — به‌معنای آموزش از صفر نیست.
۲. داده‌ی موردنیاز: `صوت + متن پیاده‌شده`.
۳. مراحل اصلی پایپلاین: پیش‌پردازش ← استخراج ویژگی ← توکن‌سازی ← آموزش ← ارزیابی.
۴. مدل‌های بزرگ مثل Whisper معمولاً به‌طور بهره‌ورتر از طریق **منجمد کردن لایه** یا **LoRA** فاین‌تیون می‌شوند.
۵. کیفیت دیتاست حوزه‌ی تخصصی معمولاً بیشتر از تعداد پارامترهایی که منجمدزدایی می‌کنید اهمیت دارد.
۶. مراقب **بیش‌برازش** باشید، به‌خصوص وقتی داده کمیاب است — که معمولاً دقیقاً همان وضعیتی است که فاین‌تیون را در وهله‌ی اول ایجاب کرده.

> **چارچوبی مهندسی که ارزش درونی کردن دارد:** بهترین استراتژی واقعی معمولاً «بزرگ‌ترین مدل را انتخاب کن» نیست — بلکه **مدل از پیش‌آموزش‌دیده + داده‌ی واقعیِ حوزه‌ی هدف + فاین‌تیون هوشمندانه = یک سیستم ASR عملی است.**

## واحد ۶ — دیتاست‌های TTS

جایی که دیتاست‌های ASR به‌صورت `صوت ← متن` هستند، دیتاست‌های TTS در **جهت مخالف** ساختار می‌یابند:
```
متن + ضبط همان متن که توسط یک انسان گفته شده
```
مدل یاد می‌گیرد: *«اگر این متن را ببینم، یک انسان چگونه آن را تلفظ می‌کند؟»*

### ساختار یک نمونه‌ی TTS

```json
{
  "text": "سلام، حالت چطوره؟",
  "audio": "hello.wav",
  "speaker": "speaker_id"
}
```

| | دیتاست ASR | دیتاست TTS |
|---|---|---|
| هدف | صوت ← متن | متن ← صوت |
| داده | `speech.wav` + `transcript.txt` | `text.txt` + `speech.wav` |

### چه چیزی یک دیتاست TTS را واقعاً خوب می‌کند

**۱. کیفیت ضبط — اینجا خیلی بیشتر از ASR اهمیت دارد**، چون مدل باید صوت طبیعی *تولید* کند، نه صرفاً صوت موجود را تفسیر کند.
```
❌ بد:   ضبط تلفنی + نویز پس‌زمینه
✅ خوب: ضبط استودیویی
```
عوامل کلیدی: نویز کم، میکروفون خوب، صدای واضح، محیط ضبط کنترل‌شده.

**۲. دقت متن پیاده‌شده — باید دقیقاً با آنچه گفته شده مطابقت داشته باشد.**
```
صوت:         «می‌خوام برم خونه»
متن پیاده‌شده: «قصد دارم به منزل بروم»   ← عدم تطابق — مشکل ایجاد می‌کند
```
اگر متن پیاده‌شده با آنچه واقعاً گفته شده مطابقت نداشته باشد، مدل یاد می‌گیرد چیزی نسبتاً متفاوت از واقعیت آکوستیکی که روی آن آموزش دیده تولید کند — این نگاشت متن-به-صدا را به شکلی خراب می‌کند که صرفاً با نگاه سرسری به فایل متن پیاده‌شده به‌سختی قابل‌تشخیص است.

**۳. تعداد گویندگان — دو نوع دیتاست اساساً متفاوت:**

| | **تک‌گوینده** | **چندگوینده** |
|---|---|---|
| ساختار | یک نفر همه‌ی جملات را می‌خواند (مثلاً ۱۰٬۰۰۰ جمله) | چند گوینده (الف، ب، ج، ...) |
| مزیت | صدای بسیار یکدست؛ آموزش آسان‌تر | انعطاف‌پذیرتر؛ چند صدای خروجی را ممکن می‌کند |
| معایب | فقط همان یک صدا را یاد می‌گیرد | آموزشش سخت‌تر است |

**۴. تنوع متنی** — مدل باید در معرض جملات کوتاه، جملات بلند، اعداد، اسامی، و علائم نگارشی قرار بگیرد:
```
«جلسه ساعت ۱۰:۳۰ شروع می‌شود.»  در برابر  «سلام.»
```
این‌ها جنبه‌های بسیار متفاوتی از آهنگ گفتار (prosody) و تلفظ را به چالش می‌کشند.

### دیتاست‌های شناخته‌شده‌ی TTS

| دیتاست | زبان/نوع | آمار کلیدی | کاربرد معمول |
|---|---|---|---|
| **LJSpeech** | انگلیسی، تک‌گوینده | حدود ۱۳٬۰۰۰ کلیپ صوتی، حدود ۲۴ ساعت | آموزش Tacotron و مدل‌های اولیه‌ی TTS |
| **LibriTTS** | انگلیسی، برگرفته از کتاب صوتی | چندگوینده، کیفیت بالاتر از LibriSpeech | آموزش مدل‌های مدرن |
| **VCTK** | انگلیسی، چندگوینده | صدها گوینده، لهجه‌های متعدد انگلیسی | تطبیق گوینده، تحقیقات شبیه‌سازی صدا |
| **Common Voice** | چندزبانه | پوشش گسترده‌ی زبانی | اصولاً ASR؛ برخی پروژه‌های TTS |

### TTS فارسی — چالش‌های خاص

ساخت TTS فارسی با دشواری‌های افزاینده روبه‌روست:
- در دسترس بودن محدود دیتاست
- تنوع محدود گوینده
- مشکلات کیفیت متن پیاده‌شده

ساخت یک سیستم TTS فارسیِ قابل‌استفاده معمولاً نیازمند ضبط حرفه‌ای، منابع متنی متنوع، و چند ساعت صوت تمیز است — هیچ میان‌بر دیتاست عمومی بزرگی معادل آنچه انگلیسی با LJSpeech/LibriTTS دارد، وجود ندارد.

### آماده‌سازی داده مختص TTS

**نرمال‌سازی متن** — تبدیل متن به شکلی که باید *خوانده* شود:
```
ورودی: «من ۱۰ سیب دارم»
خروجی: «من ده سیب دارم»
```
مدل باید بداند اعداد (و تاریخ‌ها، مخفف‌ها و غیره) واقعاً چگونه تلفظ می‌شوند، نه صرفاً شکل نوشتاری آن‌ها.

**پردازش صوت** — تغییر نرخ، حذف سکوت‌های اضافی، نرمال‌سازی صدا.

**آموزش/اعتبارسنجی/آزمون — با همان تله‌ی نشت گوینده که در ASR داشتیم:**
```
❌ بد:  آموزش: صدای علی   آزمون: صدای علی
```
مدل ممکن است به‌سادگی صدای علی را حفظ کند به‌جای یادگیری نگاشت تعمیم‌پذیر متن-به-گفتار.

### معیارهای کیفیت مختص TTS — «متن درست» کافی نیست

| معیار | سؤالی که پاسخ می‌دهد |
|---|---|
| **طبیعی بودن (Naturalness)** | آیا صوت طبیعی به نظر می‌رسد؟ |
| **شباهت (Similarity)** | آیا شبیه گوینده‌ی موردنظر به نظر می‌رسد؟ |
| **هم‌ترازی (Alignment)** | آیا متن و صوت به‌درستی همگام‌سازی شده‌اند؟ |

### پیوند دیتاست ↔ معماری

Tacotron، مثلاً، باید `متن ← Mel Spectrogram` را یاد بگیرد — پس دیتاست باید طوری ساختاردهی شود که مدل بتواند واقعاً آن تبدیل خاص را به‌طور تمیز یاد بگیرد.

### 📎 چک‌لیست حفظ‌کردنی — واحد ۶

۱. یک دیتاست TTS جفت‌های `متن + صوت` است — جهت معکوس ASR.
۲. کیفیت ضبط و دقت متن پیاده‌شده مستقیماً تعیین می‌کنند خروجی چقدر طبیعی به نظر می‌رسد.
۳. دیتاست‌ها یا **تک‌گوینده** (یکدست، ساده) یا **چندگوینده** (انعطاف‌پذیر، آموزشش سخت‌تر) هستند.
۴. دیتاست‌های شناخته‌شده: **LJSpeech، LibriTTS، VCTK، Common Voice.**
۵. متن باید پیش از آموزش **نرمال‌سازی** شود (مثلاً اعداد ← کلمات نوشتاری).
۶. در TTS، کیفیت صوت، تنوع گوینده، و هم‌ترازی متن-صوت بیشتر از حجم خام داده اهمیت دارند.

> **پیوند به ASR:** ASR از مدل می‌خواهد گفتار انسانی را *بفهمد*؛ TTS از آن می‌خواهد گفتار انسانی *تولید کند*. در هر دو، داده‌ی خوب مهم‌ترین عامل تعیین‌کننده‌ی کیفیت مدل است — انتخاب معماری در رتبه‌ی دوم و با فاصله قرار دارد.

## واحد ۶ب — مدل‌های TTS از پیش‌آموزش‌دیده

همان‌طور که ASR مدل‌های Whisper و Wav2Vec2 را دارد، TTS نیز خانواده‌ی خودش از مدل‌های از پیش‌آموزش‌دیده‌ی آماده‌به‌استفاده را دارد — ساختن یکی از صفر به‌ندرت نقطه‌ی شروع درستی است.

```
متن ← مدل TTS از پیش‌آموزش‌دیده ← گفتار
```

### چرا مدل‌های TTS از پیش‌آموزش‌دیده اهمیت دارند

ساختن TTS از صفر سخت‌تر از چیزی است که به نظر می‌رسد — مدل باید تلفظ درست کلمات، ریتم طبیعی گفتار، مکث‌گذاری، آهنگ (intonation)، پویایی بلندی صدا، و ویژگی‌های مختص گوینده را یاد بگیرد، که همگی برای یادگیری خوب به داده‌ی قابل‌توجهی نیاز دارند. مسیر عملی معمول:
```
مدل بزرگ TTS + دیتاست سفارشی کوچک ← صدای فاین‌تیون‌شده
```

### مؤلفه‌های یک سیستم TTS از پیش‌آموزش‌دیده‌ی معمولی

```
متن ← Tokenizer ← Text Encoder ← مدل آکوستیک ← Mel Spectrogram ← Vocoder ← موج صدا
```

برخی مدل‌های جدیدتر این مراحل را به‌جای نگه‌داشتن به‌عنوان مؤلفه‌های آموزشی جداگانه، در یک معماری سرتاسری (end-to-end) ادغام می‌کنند.

### خانواده‌های اصلی مدل TTS

**۱. Tacotron 2** — معماری کلاسیک و بسیار تأثیرگذار.
```
متن ← Encoder ← Attention ← Decoder ← Mel Spectrogram ← Vocoder ی WaveNet ← صوت
```
| نقاط قوت | نقاط ضعف |
|---|---|
| ✅ صدای طبیعی | ❌ کندتر |
| ✅ معماری قابل‌فهم | ❌ گاهی هم‌ترازی شکست می‌خورد |
| ✅ پایه‌ای برای بسیاری از تحقیقات بعدی TTS | ❌ تولید متن‌های طولانی می‌تواند مشکل‌ساز باشد |

**۲. FastSpeech / FastSpeech 2** — مخصوصاً برای حل مشکل سرعت Tacotron ساخته شده.
Tacotron صوت را گام‌به‌گام تولید می‌کند؛ FastSpeech بخش بسیار بیشتری را به‌صورت موازی تولید می‌کند.
```
متن ← Encoder ← پیش‌بین مدت‌زمان (Duration Predictor) ← Mel Spectrogram ← Vocoder ← گفتار
```
| نقاط قوت |
|---|
| ✅ سرعت بالا |
| ✅ کنترل بهتر روی مدت‌زمان خروجی |
| ✅ مناسب کاربردهای بی‌درنگ (Real-time) |

**۳. VITS** (*استنتاج تغییرپذیر با یادگیری تخاصمی برای تبدیل متن به گفتار*) — یک مدل مدرن، محبوب، و کاملاً سرتاسری.
```
متن ← Encoder ← نمایش نهان (Latent) ← Decoder ← صوت
```
یک Text Encoder، یک مدل متغیر نهان، یک Vocoder، و آموزش تخاصمی مبتنی بر GAN را در یک سیستم ترکیب می‌کند.
| نقاط قوت |
|---|
| ✅ کیفیت صوتی بالا |
| ✅ آموزش سرتاسری |
| ✅ کاملاً مناسب برای شبیه‌سازی صدا |

**۴. SpeechT5** — یک ترنسفورمر چندوظیفه‌ای از Hugging Face، قابل‌انجام متن←گفتار، گفتار←متن، *و* گفتار←گفتار از یک بدنه‌ی مشترک.
```
ورودی ← ترنسفورمر مشترک ← خروجی وظیفه
```
مزیت: یک مدل، چند وظیفه را پوشش می‌دهد.

**۵. Bark** — یک مدل تولید صوت که قادر به تولید گفتار، موسیقی ساده، جلوه‌های صوتی، و **صداهای غیرکلامی** است:
```
«خنده» ← 😂 (صدای خنده‌ی تولیدشده)
```

**۶. XTTS** — متمرکز بر پشتیبانی چندزبانه و شبیه‌سازی صدا:
```
ورودی:  متن + نمونه‌ی کوتاه صدای گوینده
خروجی: گفتار با همان صدا
```

### جدول مقایسه‌ی مدل‌ها

| مدل | نوع معماری | ویژگی متمایزکننده |
|---|---|---|
| **Tacotron 2** | Encoder-Decoder | کلاسیک، صدای طبیعی |
| **FastSpeech 2** | غیرخودبازگشتی | سریع |
| **VITS** | سرتاسری + GAN | کیفیت بالا |
| **SpeechT5** | ترنسفورمر | چندوظیفه‌ای |
| **Bark** | تولیدی | تولید متنوع صوت، صداهای غیرگفتاری |
| **XTTS** | چندزبانه | شبیه‌سازی صدا |

### استفاده از یک مدل TTS از پیش‌آموزش‌دیده در Hugging Face

```python
from transformers import pipeline

pipe = pipeline("text-to-speech", model="model_name")
output = pipe("سلام، حالت چطوره؟")
# {"audio": waveform, "sampling_rate": 22050}
```

### فاین‌تیون کردن یک مدل TTS برای یک صدای سفارشی

```
TTS از پیش‌آموزش‌دیده (مثلاً VITS) + دیتاست صدای سفارشی (متن + صوت گوینده) ← مدل TTS سفارشی
```
اهداف رایج: یک صدای فارسی، یک صدای برندشده، یا صدای یک فرد خاص.

### شبیه‌سازی صدا (Voice Cloning) — مکانیزم کلیدی فعال‌کننده

TTS مدرن فراتر از «این متن را با صدای بلند بخوان» می‌رود — هدف تبدیل به *«صوتی تولید کن که شبیه این فرد خاص باشد»* می‌شود. این معمولاً نیازمند این است:
```
صوت گوینده ← Speaker Encoder ← Embedding گوینده ← مدل TTS
```

### ASR در برابر TTS، از منظر مدل از پیش‌آموزش‌دیده

| | ASR | TTS |
|---|---|---|
| ورودی | صوت | متن |
| خروجی | متن | صوت |
| مدل‌های پرچمدار | Whisper، Wav2Vec2 | VITS، FastSpeech، Tacotron |
| دشواری اصلی | فهمیدن گفتار | تولید گفتار طبیعی |

### 📎 چک‌لیست حفظ‌کردنی — واحد ۶ب

۱. مدل‌های TTS از پیش‌آموزش‌دیده به شما اجازه می‌دهند بدون آموزش از صفر، از متن به گفتار برسید.
۲. پایپلاین کلاسیک: `متن ← مدل آکوستیک ← اسپکتروگرام ← Vocoder ← صوت`.
۳. مدل‌های کلیدی: **Tacotron 2** (پایه‌ی کلاسیک) · **FastSpeech** (سرعت) · **VITS** (کیفیت، سرتاسری) · **SpeechT5** (چندوظیفه‌ای) · **XTTS** (چندزبانه، شبیه‌سازی صدا).
۴. صداهای سفارشی/برندشده معمولاً از طریق **فاین‌تیون** یا **Speaker Embedding** ساخته می‌شوند، نه آموزش از صفر.
۵. TTS خوب فقط کلمات را درست تلفظ نمی‌کند — باید طبیعی به نظر برسد، لحن مناسب داشته باشد، و شبیه گفتار واقعی انسان باشد.

> **پیوند به عقب:** ASR یاد می‌گیرد `صوت ← نمایش ← متن`؛ TTS یاد می‌گیرد `متن ← نمایش ← صوت`. هر دو نهایتاً همان چیز زیربنایی را از جهت‌های مخالف یاد می‌گیرند: رابطه‌ی بین زبان و سیگنال آکوستیک.

## واحد ۶ج — فاین‌تیون SpeechT5: یک راهنمای گام‌به‌گام

این بخش SpeechT5 را برای یک صدا، زبان، یا حوزه‌ی خاص شخصی‌سازی می‌کند — یک نسخه‌ی مشخص و گام‌به‌گام از مفاهیم فاین‌تیونی که پیش‌تر معرفی شد.

### SpeechT5 چیست

یک مدل چندوظیفه‌ای مبتنی بر ترنسفورمر از Hugging Face که قادر به این‌هاست:
- متن ← گفتار (TTS)
- گفتار ← متن (ASR)
- گفتار ← گفتار

```
متن ← Encoder-Decoder ی SpeechT5 ← Mel Spectrogram ← Vocoder ← موج صدا
```

### چرا آن را فاین‌تیون می‌کنید

مدل پایه روی داده‌ی عمومی آموزش دیده. ممکن است نیاز داشته باشید:
- یک صدای خاص تولید کند
- از یک زبان خاص بهتر پشتیبانی کند (مثلاً `TTS انگلیسی ← TTS فارسی`)
- در یک حوزه‌ی خاص بهتر عمل کند (مثلاً `← صدای پشتیبانی مشتری`)

### دیتاستی که فاین‌تیون SpeechT5 به آن نیاز دارد

```json
{"text": "سلام به همه", "audio": "speaker001.wav"}
```
ورودی مدل: «سلام به همه» ← هدف مدل: ضبط انسانی همان جمله‌ی دقیق.

### نقش Speaker Embedding

ویژگی تعریف‌کننده‌ی SpeechT5 مشروط کردن تولید بر اساس اینکه **چه کسی** باید صحبت کند:
```
صوت گوینده ← Speaker Encoder ← Embedding گوینده ← SpeechT5 ← صدای تولیدشده
```

| بدون Speaker Embedding | با Speaker Embedding |
|---|---|
| ممکن است یک صدای پیش‌فرض/عمومی تولید شود | متن با صدای یک گوینده‌ی خاص و انتخاب‌شده رندر می‌شود |

### فرایند پنج‌مرحله‌ای فاین‌تیون

**گام ۱ — آماده‌سازی دیتاست.**
```
dataset/
 ├── audio/
 │     ├── 001.wav
 │     ├── 002.wav
 └── metadata.csv       # «001.wav | سلام دنیا» / «002.wav | حالت چطوره»
```

**گام ۲ — پردازش متن.** توکن‌سازی: «سلام» ← `[39, 12, 5, 5, 8]`.

**گام ۳ — پردازش صوت.** یکسان‌سازی نرخ نمونه‌برداری، سپس تبدیل به Mel Spectrogram — مدل مستقیماً **Mel Spectrogram** را پیش‌بینی می‌کند، نه موج خام را:
```
موج صدا ← Mel Spectrogram
```

**گام ۴ — آموزش.** ورودی = `متن + Embedding گوینده`. مدل یک `Mel Spectrogram پیش‌بینی‌شده` تولید می‌کند، که با `Mel واقعی` مقایسه می‌شود تا loss محاسبه شود، و وزن‌ها بر همان اساس به‌روزرسانی می‌شوند.

**گام ۵ — اعمال Vocoder.**
```
SpeechT5 ← Mel Spectrogram ← Vocoder (معمولاً HiFi-GAN) ← صوت
```

### مسئله‌ی عملی مرکزی: کمبود داده

TTS به‌طور غیرمعمولی به کیفیت داده حساس است:
> **۱۰ ساعت صوت تمیز و تک‌گوینده می‌تواند از ۱۰۰ ساعت صوت پرنویز و چندگوینده بهتر عمل کند.**

### چرا Data Collator لازم است

جملات به‌شدت در طول متفاوت‌اند:
```
«سلام»                                ← کوتاه
«این یک جمله‌ی بسیار طولانی است...»    ← بلند
```
Batch کردن به شکل‌های Tensor یکسان نیاز دارد، پس دنباله‌ها باید تا یک طول مشترک **بالشتک‌گذاری (Pad)** شوند — **Data Collator** این کار را به‌طور خودکار انجام می‌دهد.

### ارزیابی یک مدل TTS فاین‌تیون‌شده

ارزیابی TTS ذاتاً سخت‌تر از ارزیابی ASR است، چون هیچ حقیقت پایه‌ی ساده‌ی مبتنی بر فاصله‌ی ویرایشی برای «چقدر طبیعی به نظر رسید» وجود ندارد:

| معیار | چه چیزی را اندازه می‌گیرد |
|---|---|
| **MOS** (میانگین امتیاز نظرسنجی) | ارزیابان انسانی طبیعی بودن را نمره می‌دهند، معمولاً از ۱ (خیلی بد) تا ۵ (کاملاً طبیعی) |
| **شباهت گوینده** | آیا صدای تولیدشده با گوینده‌ی هدف مطابقت دارد؟ |
| **قابل‌فهم بودن** | آیا کلمات واقعاً قابل‌فهم هستند؟ |

### فاین‌تیون کامل در برابر جزئی برای مدل‌های بزرگ TTS

برای مدل‌های بزرگ، به‌روزرسانی هر وزن اغلب لازم نیست:

**منجمد کردن:**
```
Encoder ❄️  منجمد
Decoder 🔥  آموزش‌دیده
```
فواید: حافظه‌ی کمتر، آموزش سریع‌تر.

**روش‌های PEFT مثل LoRA:** به‌جای میلیاردها پارامتر، یک آداپتور کوچک آموزش داده می‌شود.

### فاین‌تیون ASR در برابر فاین‌تیون TTS، کنار هم

| | ASR | TTS |
|---|---|---|
| ورودی | صوت | متن |
| خروجی | متن | صوت |
| هدف | درک | تولید |
| دیتاست | صوت + متن پیاده‌شده | متن + گفتار |
| معیار | WER | MOS |

### مدل ذهنی کامل فاین‌تیون SpeechT5

```
متن
 │
 ▼
Tokenizer
 │
 ▼
SpeechT5 ──── + Embedding گوینده
 │
 ▼
Mel Spectrogram
 │
 ▼
Vocoder ی HiFi-GAN
 │
 ▼
گفتار
```

### 📎 چک‌لیست حفظ‌کردنی — واحد ۶ج

۱. SpeechT5 یک ترنسفورمر چندمنظوره برای وظایف صوتی (ASR، TTS، و بیشتر) است.
۲. فاین‌تیون TTS نیازمند جفت‌های `متن + ضبط صوتی` است.
۳. SpeechT5 از **Speaker Embedding** برای کنترل هویت صدا در خروجی استفاده می‌کند.
۴. مدل معمولاً این مسیر را یاد می‌گیرد: `متن ← Mel Spectrogram ← Vocoder ← صوت`.
۵. فاین‌تیون یعنی تطبیق یک مدل عمومی با یک زبان، حوزه، یا صدای خاص — نه ساختن از صفر.
۶. در TTS، **کیفیت داده‌ی صوتی حتی بیشتر از حجم داده اهمیت دارد.**

> این واحد عملاً آخرین بلوک ساختمانی بزرگ پیش از ورود به سیستم‌های کامل هوش مصنوعی صوتی است: **ASR برای فهمیدن، یک مدل زبانی برای استدلال، و TTS برای صحبت کردن** — که دقیقاً همان‌جایی است که [واحد ۷](#واحد-۷--کنار-هم-گذاشتن-همه‌چیز) ادامه می‌یابد.

## واحد ۷ — کنار هم گذاشتن همه‌چیز

این واحد ترکیب کل دوره است. تا اینجا، به‌صورت جداگانه این‌ها را یاد گرفته‌اید:

```
ASR                     ← گفتار ← متن
TTS                     ← متن ← گفتار
دسته‌بندی صوت           ← صوت ← برچسب
معماری‌های ترنسفورمر
فاین‌تیون
```

هدف اکنون: **ترکیب این قطعات در سیستم‌های صوتی واقعی و کاربردی.** سه پروژه‌ی کلاسیک محور این واحد هستند:
۱. ترجمه‌ی گفتار-به-گفتار
۲. ساختن یک دستیار صوتی
۳. پیاده‌سازی یک جلسه

### ۱. ترجمه‌ی گفتار-به-گفتار

**هدف:** تبدیل گفتار به یک زبان، به گفتار به زبان دیگر.
```
ورودی:  🎤 گفتار آلمانی
خروجی: 🔊 گفتار انگلیسی
```

**معماری آبشاری کلاسیک — سه مدل که به‌ترتیب زنجیر شده‌اند:**
```
گفتار ← ASR ← متن ← مدل ترجمه ← متن ترجمه‌شده ← TTS ← گفتار
```
یعنی، **تشخیص گفتار + ترجمه‌ی ماشینی + سنتز گفتار.**

**مثال حل‌شده:**
```
ورودی:        "Bonjour, comment allez-vous?"
مرحله‌ی ASR:   Bonjour, comment allez-vous?
ترجمه:        سلام، حالت چطوره؟
مرحله‌ی TTS:   🔊 «سلام، حالت چطوره؟»
```

**معماری‌های جدیدتر و مستقیم‌تر** (مثلاً **SeamlessM4T**) کاملاً از متن میانی صرف‌نظر می‌کنند:
```
گفتار ← مدل گفتار-به-گفتار ← گفتار
```

**معامله‌ی میان این دو رویکرد:**

| | آبشاری (۳ مدل جدا) | گفتار-به-گفتار مستقیم |
|---|---|---|
| حفظ لحن/آهنگ گفتار | ضعیف‌تر — در هر واسط متنی از دست می‌رود | ✅ بهتر حفظ می‌شود |
| سرعت | کندتر (سه مرحله‌ی سریالی) | ✅ سریع‌تر |
| طبیعی بودن | پایین‌تر | ✅ بالاتر |
| قابلیت ماژولار بودن | ✅ هر مؤلفه به‌طور مستقل قابل‌تعویض/دیباگ | دیباگ یا تعویض تک‌تک قطعات دشوارتر است |

هیچ رویکردی به‌طور قطعی برتر نیست — قابلیت ماژولار بودن رویکرد آبشاری (جایگزین کردن یک مدل ترجمه‌ی بهتر بدون بازآموزی چیز دیگری) یک مزیت مهندسی واقعی است، هرچند رویکرد مستقیم معمولاً طبیعی‌تر به نظر می‌رسد.

### ۲. ساختن یک دستیار صوتی

این همان الگوی Siri / Alexa / Google Assistant است:
```
🎤 گفتار کاربر ← ASR ← مدل زبانی ← TTS ← 🔊 پاسخ
```

**مرحله‌ی ۱ — تشخیص گفتار.**
```
کاربر: «هوا امروز چطوره؟»
ASR:  صوت ← متن ← «هوا امروز چطوره؟»
```

**مرحله‌ی ۲ — درک/استدلال.** یک مدل زبانی قصد را تجزیه و پارامترها را استخراج می‌کند:
```
Intent:   weather_query
Location: تهران
```

**مرحله‌ی ۳ — تولید گفتار.**
```
متن پاسخ: «امروز هوا آفتابی است.»
TTS: متن ← گفتار
```

### چالش‌های واقعی تولید — جایی که تلاش مهندسی واقعی صرف می‌شود

**تأخیر.** کاربران مکث‌های چند ثانیه‌ای را تحمل نمی‌کنند — هر مؤلفه (ASR، مدل زبانی، TTS) باید آن‌قدر سریع باشد که پایپلاین *ترکیبی* هنوز پاسخگو به نظر برسد.

**استریمینگ.** سیستم نباید منتظر یک جمله‌ی کامل بماند تا پردازش را شروع کند:
```
کاربر (هنوز در حال صحبت): «می‌خوام بدونم...»
                              ↓
                    مدل از قبل در حال پردازش
```
```
جریان صوتی ← ASR استریم ← مدل زبانی ← TTS استریم
```

**تشخیص عبارت بیدارباش (Wake Word).** سیستم دائماً گوش می‌دهد اما فقط با یک عبارت محرک (مثلاً «هی گوگل») کاملاً فعال می‌شود — یک دسته‌بند کوچک، ارزان و همیشه-روشن، دروازه‌ی یک پایپلاین پایین‌دستی بسیار سنگین‌تر است، به‌طوری‌که مؤلفه‌های گران فقط زمانی اجرا می‌شوند که واقعاً لازم باشد.

### ۳. پیاده‌سازی یک جلسه

**هدف:** تبدیل صوت جلسه به متن قابل‌جستجو و منتسب به گوینده.
```
ورودی:  صوت جلسه‌ی ۲ ساعته
خروجی: علی: باید معماری را بهبود دهیم...
        سارا: موافقم...
```

**معماری سیستم:**
```
صوت جلسه ← Speaker Diarization ← ASR ← متن پیاده‌شده ← خلاصه/جستجو
```

**Speaker Diarization واقعاً چیست:** تعیین *چه کسی چه زمانی صحبت کرده*، کاملاً مستقل از *چه چیزی* گفته شده.
```
صوت:    [شخص الف] [شخص ب] [شخص الف]
خروجی: گوینده‌ی ۱: سلام
        گوینده‌ی ۲: سلام
        گوینده‌ی ۱: بیایید شروع کنیم
```

### پایپلاین کامل پیاده‌سازی جلسه، مرحله‌به‌مرحله

```
۱. پردازش صوت                      ← حذف نویز، بخش‌بندی فایل
۲. تشخیص فعالیت صوتی (VAD)          ← تشخیص گفتار در برابر سکوت
۳. Speaker Diarization              ← تعیین اینکه چه کسی چه زمانی صحبت کرده
۴. ASR                              ← گفتار ← متن
۵. پس‌پردازش                        ← اصلاح متن، بازیابی علائم نگارشی، خلاصه‌سازی
```

**تشخیص فعالیت صوتی، به‌طور ملموس:**
```
جدول زمانی صوت: [سکوت] [گفتار] [سکوت] [گفتار]
```
VAD چیزی است که به سیستم اجازه می‌دهد از هدر دادن محاسبات برای پیاده‌سازی سکوت اجتناب کند، و کمک می‌کند فایل پیش از شروع Diarization به‌طور تمیز بخش‌بندی شود.

### چگونه هر واحد این دوره به هم متصل می‌شود

```
                            هوش مصنوعی صوت
                    ┌──────────────────┴──────────────────┐
                    ▼                                       ▼
                  ASR                                     TTS
             گفتار ← متن                              متن ← گفتار
                    │                                       │
                    └───────────────────┬───────────────────┘
                                        ▼
                              کاربردهای صوتی
```

### سه سیستم واقعی، نگاشت‌شده به پایپلاین مؤلفه‌های خود

**ChatGPT Voice / دستیارهای صوتی عمومی:**
```
صدای کاربر ← ASR ← مدل زبانی ← TTS ← صدای دستیار
```

**مترجم همزمان:**
```
گفتار زبان الف ← ASR ← ترجمه ← TTS ← گفتار زبان ب
```

**هوش مصنوعی جلسه:**
```
صوت ← VAD ← Diarization ← ASR ← مدل خلاصه‌سازی ← یادداشت‌ها
```

### 📎 چک‌لیست حفظ‌کردنی — واحد ۷

۱. محصولات صوتی واقعی **پایپلاین‌هایی از چند مدل تخصصی** هستند — تقریباً هرگز یک مدل یکپارچه‌ی تنها.
۲. یک مترجم گفتار = **ASR + ترجمه + TTS.**
۳. یک دستیار صوتی = **ASR + مدل زبانی + TTS.**
۴. پیاده‌سازی جلسه نیازمند **VAD + Speaker Diarization + ASR** است (به‌علاوه‌ی پس‌پردازش).
۵. چالش‌های سیستم واقعی که در محک‌های مدل ایزوله ظاهر نمی‌شوند: **تأخیر، استریمینگ، حافظه، دقت، مدیریت چندگوینده.**

### نمودار خلاصه‌ی کل دوره

```
                       صوت ورودی
                           │
                 ┌─────────┴─────────┐
                 ▼                   ▼
                ASR              دسته‌بندی
                 │
                 ▼
                متن
                 │
                 ▼
             مدل زبانی / NLP
                 │
                 ▼
                TTS
                 │
                 ▼
           گفتار خروجی
```

**یک جمله‌ای که کل این حوزه را خلاصه می‌کند:**
> یک سیستم کامل هوش مصنوعی صوتی، ترکیبی از **پردازش سیگنال + یادگیری عمیق + ترنسفورمرها + NLP** است — هیچ مدل تنهایی همه‌کاره نیست، و مهارت مهندسی واقعی در ترکیب مؤلفه‌های تخصصی درست و مدیریت معامله‌های تأخیر/دقت/استحکام در هر درزی که به هم متصل می‌شوند نهفته است.

## 📊 جدول‌های مقایسه‌ی میان‌بخشی

یک صفحه‌ی مرجع تکی برای مقایسه‌هایی که در سراسر این راهنما تکرار می‌شوند.

### معماری بر اساس وظیفه

| وظیفه | معماری معمول |
|---|---|
| ASR | CTC یا Seq2Seq |
| دسته‌بندی صوت | Encoder + سر دسته‌بند |
| TTS | Seq2Seq / تولیدی (مدل آکوستیک + Vocoder) |
| تولید صوت | مدل‌های تولیدی |

### دو خانواده‌ای که هر مدل صوتی به آن‌ها تعلق دارد

| | تحلیلی (صوت ← برچسب/متن) | تولیدی (متن ← صوت) |
|---|---|---|
| مثال‌ها | ASR، دسته‌بندی، تشخیص گوینده/احساس | TTS، تولید صوت |
| عامل دشواری | باید یک سیگنال ثابت را درست *تفسیر* کند | باید یک سیگنال باورپذیر را از صفر *سنتز* کند |
| خانواده‌ی معیار معمول | WER / دقت / F1 | MOS / شباهت گوینده |

### CTC در برابر Seq2Seq (به‌طور کامل برای مرجع تکرار شده)

| | CTC | Seq2Seq (Encoder-Decoder) |
|---|---|---|
| معماری | Encoder + خطی | Encoder + Decoder |
| تولید خروجی | مستقیم، فریم‌به‌فریم | گام‌به‌گام (خودبازگشتی) |
| هم‌ترازی | با قانون فشرده‌سازی حل می‌شود | از طریق Attention حل می‌شود |
| سرعت | سریع‌تر | کندتر |
| انعطاف‌پذیری | کمتر | بیشتر |
| ترجمه | ضعیف | قوی |
| مدل پرچمدار | Wav2Vec2 | Whisper |

### ASR در برابر TTS

| | ASR | TTS |
|---|---|---|
| ورودی | صوت | متن |
| خروجی | متن | صوت |
| هدف | فهمیدن | تولید کردن |
| شکل دیتاست | صوت + متن پیاده‌شده | متن + ضبط گفتار |
| معیار اصلی | WER | MOS |
| مدل‌های پرچمدار | Whisper، Wav2Vec2 | VITS، FastSpeech، Tacotron، SpeechT5 |

### راهنمای سریع انتخاب مدل

| نیاز | مدل پیشنهادی |
|---|---|
| ASR سریع و سبک | Wav2Vec2 |
| ASR چندزبانه‌ی قوی | Whisper |
| ترجمه‌ی گفتار | Whisper / SeamlessM4T |
| سیستم صوتی چندوظیفه‌ای | SpeechT5 |
| TTS کلاسیک و طبیعی | Tacotron 2 |
| TTS بی‌درنگ | FastSpeech 2 |
| بالاترین کیفیت TTS سرتاسری | VITS |
| شبیه‌سازی صدا چندزبانه | XTTS |
| تولید صوت غیرگفتاری | Bark، MusicGen، AudioGen |

### راهنمای سریع معیارها

| معیار | وظیفه | فرمول/تعریف | کمتر یا بیشتر بهتر است |
|---|---|---|---|
| WER | ASR | (S+D+I)/N | کمتر |
| CER | ASR (سطح کاراکتر) | مشابه WER، واحد کاراکتر | کمتر |
| RTF | ASR (سرعت) | زمان پردازش / مدت‌زمان صوت | کمتر (< ۱ = سریع‌تر از زمان واقعی) |
| تأخیر | ASR / دستیارهای صوتی | زمان تا اولین پاسخ | کمتر |
| MOS | TTS | امتیاز انسانی، ۱ تا ۵ | بیشتر |
| شباهت گوینده | TTS | تطابق با صدای هدف | بیشتر |
| قابل‌فهم بودن | TTS | آیا کلمات قابل‌فهم‌اند؟ | بیشتر |

## 📖 واژه‌نامه

<details>
<summary><b>برای مشاهده کلیک کنید — بیش از ۳۰ اصطلاح، با تعریف دقیق و مختصر</b></summary>

<br>

| اصطلاح | تعریف |
|---|---|
| **موج صدا (Waveform)** | نمایش دیجیتال خام صدا: دنباله‌ای از مقادیر دامنه که در طول زمان نمونه‌برداری شده‌اند. |
| **نرخ نمونه‌برداری (Sampling Rate)** | تعداد نمونه‌های دامنه که در هر ثانیه ضبط می‌شوند (مثلاً ۱۶٬۰۰۰ هرتز = ۱۶٬۰۰۰ نمونه در ثانیه). |
| **تغییر نرخ نمونه‌برداری (Resampling)** | تبدیل صوت از یک نرخ نمونه‌برداری به نرخ دیگر؛ تعداد نمونه را تغییر می‌دهد، نه سرعت پخش را. |
| **Mel Spectrogram** | یک نمایش زمان-فرکانس از صوت، مقیاس‌شده متناسب با ادراک پرده‌ی صوتی انسان؛ یک هدف میانی رایج برای مدل‌های TTS. |
| **MFCC** | ضرایب کپسترال فرکانس-مل — ویژگی‌های دستی-مهندسی‌شده‌ی کلاسیک صوت، که تا حد زیادی جای خود را به ویژگی‌های یادگرفته‌شده‌ی CNN در مدل‌های مدرن داده است. |
| **Feature Extractor** | مؤلفه‌ای که موج خام را به یک نمایش ساختاریافته تبدیل می‌کند که یک مدل می‌تواند مصرف کند. |
| **Embedding** | یک نمایش برداری چگال از یک واحد گسسته (یک کلمه، یک فریم صوت و غیره). |
| **Self-Attention** | مکانیزمی که به مدل اجازه می‌دهد اهمیت هر موقعیت دیگر در یک دنباله را هنگام پردازش یک موقعیت مشخص، وزن‌دهی کند. |
| **Encoder** | بخشی از مدل که مسئول *فهمیدن* ورودی است (صوت/متن ← معنا). |
| **Decoder** | بخشی از مدل که مسئول *تولید* خروجی است، معمولاً گام‌به‌گام (معنا ← متن/صوت). |
| **CTC (دسته‌بندی زمانی وابسته‌محور)** | یک روش آموزش/رمزگشایی که مسئله‌ی هم‌ترازی صوت-به-متن را بدون نیاز به برچسب‌های زمانی سطح فریم، از طریق یک توکن خالی و قوانین فشرده‌سازی حل می‌کند. |
| **توکن خالی (Blank Token)** | نماد ویژه‌ی CTC به معنای «اینجا کاراکتر جدیدی نیست»؛ ابهام میان کاراکترهای تکراری و کاراکتر تکی و ممتد را برطرف می‌کند. |
| **Seq2Seq (دنباله‌به‌دنباله)** | معماری‌ای که یک دنباله‌ی ورودی را از طریق یک جفت Encoder-Decoder به یک دنباله‌ی خروجی نگاشت می‌کند، بدون نیاز به ترتیب یا طول یکسان. |
| **رمزگشایی حریصانه (Greedy Decoding)** | انتخاب تک‌گزینه‌ی با بالاترین احتمال در هر گام تولید؛ سریع اما همیشه بهینه‌ی کلی نیست. |
| **جستجوی پرتویی (Beam Search)** | ردیابی چند دنباله‌ی کاندید همزمان، سپس انتخاب بهترین کلی؛ دقیق‌تر اما سنگین‌تر. |
| **WER (نرخ خطای کلمه)** | (جایگزینی‌ها + حذف‌ها + افزایش‌ها) / کل کلمات مرجع — معیار استاندارد دقت ASR. |
| **CER (نرخ خطای کاراکتر)** | همان فرمول WER، محاسبه‌شده در سطح کاراکتر. |
| **RTF (فاکتور زمان-واقعی)** | زمان پردازش / مدت‌زمان صوت؛ RTF < ۱ به معنای سریع‌تر از زمان واقعی است. |
| **تأخیر (Latency)** | زمان سپری‌شده بین ورودی و تولید اولین خروجی قابل‌استفاده‌ی سیستم. |
| **Pooling** | فشرده‌سازی یک دنباله‌ی با طول متغیر از بردارها در یک بردار با طول ثابت (مثلاً از طریق میانگین‌گیری)، که برای دسته‌بندی روی صوت با مدت‌زمان متغیر لازم است. |
| **فاین‌تیون (Fine-tuning)** | ادامه‌ی آموزش یک مدل از قبل پیش‌آموزش‌دیده روی داده‌ی مختص یک حوزه، به‌جای آموزش از صفر. |
| **یادگیری خودنظارتی (Self-Supervised Learning)** | پیش‌آموزش روی داده‌ی بدون برچسب، با وادار کردن مدل به پیش‌بینی بخش‌هایی از ورودی خودش؛ استفاده‌شده توسط Wav2Vec2 برای یادگیری از صوت خام پیش از دخالت هر متن پیاده‌شده‌ای. |
| **منجمد کردن لایه (Layer Freezing)** | ثابت نگه‌داشتن وزن‌های برخی لایه‌های مدل هنگام فاین‌تیون، برای کاهش هزینه‌ی محاسباتی و حافظه. |
| **PEFT (فاین‌تیون بهره‌ور از نظر پارامتر)** | استراتژی‌های فاین‌تیون (مثلاً LoRA) که تعداد کمی پارامتر جدید/آداپتور را به‌جای کل مدل آموزش می‌دهند. |
| **LoRA (تطبیق رتبه‌ی پایین)** | یک تکنیک خاص PEFT که ماتریس‌های کوچک قابل‌آموزشِ با رتبه‌ی پایین را به یک مدل منجمد تزریق می‌کند. |
| **افزایش داده (Data Augmentation)** | گسترش مصنوعی تنوع مؤثر یک دیتاست (افزودن نویز، تغییر سرعت/پرده‌ی صدا و غیره) بدون جمع‌آوری داده‌ی واقعی جدید. |
| **نشت داده (Data Leakage)** | زمانی که اطلاعاتی که نباید در زمان آزمون در دسترس باشد (مثلاً یک گوینده که در آموزش هم حضور دارد)، عملکرد ظاهری مدل را به‌طور مصنوعی بالا می‌برد. |
| **استریمینگ (دیتاست)** | بارگذاری و پردازش نمونه‌های دیتاست به‌صورت درخواستی (on demand) به‌جای دانلود کل دیتاست از پیش. |
| **Vocoder** | مؤلفه‌ای که یک نمایش آکوستیک میانی (مثلاً Mel Spectrogram) را به یک موج صدای واقعاً قابل‌پخش تبدیل می‌کند. HiFi-GAN یک مثال رایج است. |
| **Speaker Embedding** | یک نمایش برداری از ویژگی‌های صدای یک گوینده‌ی خاص، که برای شرطی کردن خروجی TTS به سمت آن صدا استفاده می‌شود. |
| **شبیه‌سازی صدا (Voice Cloning)** | تولید گفتار با صدای یک فرد هدف خاص، معمولاً با استفاده از یک Speaker Embedding مشتق‌شده از یک نمونه‌ی مرجع کوتاه. |
| **MOS (میانگین امتیاز نظرسنجی)** | امتیازی از طبیعی بودن که توسط انسان به گفتار سنتزشده داده می‌شود، معمولاً در مقیاس ۱ تا ۵. |
| **Speaker Diarization** | تعیین اینکه *چه کسی* در یک ضبط صوتی *چه زمانی* صحبت کرده، مستقل از پیاده‌سازی *چه چیزی* گفته شده. |
| **VAD (تشخیص فعالیت صوتی)** | تشخیص اینکه کدام بخش‌های یک جریان صوتی شامل گفتار هستند در برابر سکوت. |
| **عبارت بیدارباش (Wake Word)** | یک عبارت محرک (مثلاً «هی گوگل») که یک سیستم صوتیِ معمولاً بی‌کار و همیشه-گوش‌به‌زنگ را فعال می‌کند. |
| **نرمال‌سازی متن (Text Normalization)** | تبدیل متن نوشتاری به شکلی که باید گفته شود (مثلاً «۱۰» ← «ده»). |

</details>

## ✅ نکات کلیدی (کل دوره، در یک صفحه)

۱. **هر نمونه‌ی یادگیری ماشین صوتی** به‌صورت `audio.{path, array, sampling_rate}` + یک برچسب مختص وظیفه است — مدل فقط `array` را می‌بیند، هرگز خود فایل را.
۲. **تغییر نرخ به نرخ آموزشی مدل (معمولاً ۱۶ کیلوهرتز) الزامی است**، و تعداد نمونه را تغییر می‌دهد، نه سرعت یا پرده‌ی صدای ادراک‌شده را.
۳. **استریمینگ دسترسی تصادفی را با هزینه‌ی دیسک/RAM تقریباً صفر معامله می‌کند** — بالای چند صد گیگابایت ضروری، پایین‌تر از آن غیرضروری است.
۴. **بدنه‌ی جهانی** این است: `موج صدا ← ویژگی‌ها ← Encoder ترنسفورمر ← سر وظیفه ← پیش‌بینی`؛ فقط سر بین وظایف تغییر می‌کند.
۵. **فقط-Encoder در برابر Encoder-Decoder، پیش‌بین‌ترین انشعاب معماری است** — این را اول درباره‌ی هر مدل جدیدی که می‌بینید بپرسید.
۶. **CTC در برابر Seq2Seq یک معامله‌ی سرعت/انعطاف‌پذیری است**: CTC سریع اما دقیقاً یکنواست؛ Seq2Seq کندتر اما می‌تواند ترتیب را عوض کند، ترجمه کند، و آزادانه تولید کند.
۷. **WER = (S+D+I)/N لازم است اما کافی نیست** — RTF، تأخیر، و استحکام نهایتاً قابلیت استفاده در تولید را تعیین می‌کنند، نه رتبه در جدول امتیاز محک.
۸. **کیفیت دیتاست در عمل بر انتخاب معماری غالب است** — یک دیتاست کوچک‌تر، تمیزتر، خوب‌برچسب‌گذاری‌شده، و منطبق با استقرار، به‌طور مرتب از یک دیتاست بزرگ‌تر و پرنویز بهتر عمل می‌کند، هم در ASR و به‌خصوص در TTS.
۹. **نشت هویت گوینده میان بخش‌های آموزش/آزمون** یک اشتباه کلاسیک و مختص صوت است که به‌طور خاموش دقت ظاهری را بالا می‌برد — به‌طور صریح مراقب آن باشید.
۱۰. **در TTS و صوت تولیدی، کیفیت داده بسیار قاطعانه‌تر از حجم داده بر نتیجه غالب است** تا در ASR — چون مدل باید ظرافت آکوستیک را بازتولید کند، و هر نویزی در داده‌ی آموزشی به یک آرتیفکت در هر نمونه‌ی تولیدشده تبدیل می‌شود.
۱۱. **محصولات واقعی پایپلاین هستند، نه یک مدل تنها** — ترجمه‌ی گفتار-به-گفتار، دستیارهای صوتی، و پیاده‌سازی جلسه، هرکدام ۳+ مؤلفه‌ی تخصصی را ترکیب می‌کنند، و سخت‌ترین چالش‌های مهندسی (تأخیر، استریمینگ، Diarization، دروازه‌بانی عبارت بیدارباش) در درزهای بین مؤلفه‌ها زندگی می‌کنند، نه درون هیچ‌کدام از آن‌ها به‌تنهایی.
۱۲. **بزرگ‌تر خودکار به معنای بهتر نیست** — انتخاب مدل یک معامله‌ی واقعی بین دقت، تأخیر، حافظه، و سخت‌افزار هدف است، نه یک بهینه‌سازی تک‌بُعدی.
